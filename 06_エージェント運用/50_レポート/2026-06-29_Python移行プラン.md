## 1. いま何が起きてるの？（現状）

ひとことで言うと「**全部のスクリプトが、macOSに最初から入ってる古いPython（3.9.6）で動いてる**」状態。

- いま動いてるPython: **Python 3.9.6**（実体は `/usr/bin/python3`、macOS標準のやつ）
- このPythonは**もうサポート終了（EOL）**してて、これから先「セキュリティの穴」や「新しいライブラリが入らない」が増えていく。
- 一番こわいのは、**Google系のライブラリ（YouTube売上の取得などに使う）が新しくなると、古いPythonでは動かなくなって認証が止まる**こと。これがP36の心配ごと。

### いまPythonを呼んでる場所（3種類）

| 呼び方 | 場所 | どう書いてあるか |
|---|---|---|
| ① launchdから直接 | 3つのジョブ | プラスト内に `<string>/usr/bin/python3</string>` とフルパスで固定書き |
| ② シェルスクリプト経由 | run_*.sh など多数 | `python3 ...`（PATH頼み＝今は3.9.6に当たる）と `/usr/bin/python3 ...` が混在 |
| ③ スクリプトの1行目 | .pyファイル47個 | `#!/usr/bin/env python3`（PATH頼み） |

**launchdが直接 `/usr/bin/python3` を名指ししてるジョブ（ここだけフルパス固定なので、切替時に書き換えが必要）:**
- `com.claude.freee-uncleared-monitor` → `~/.claude/scripts/freee-uncleared-monitor.py`
- `com.claude.weekly-stocktake` → `~/.claude/scripts/weekly_stocktake.py`
- `com.claude.youtube-revenue` → `~/.config/youtube-revenue/fetch_daily.py`（★Google認証を使う＝P36の本丸）

その他のlaunchdジョブ（20個以上）は、シェルスクリプト（.sh）を呼んでいて、その中で `python3`/`/usr/bin/python3` を使う形。

### 「認証が止まると困る」Google系を使ってるスクリプト

- `~/.config/youtube-revenue/` … `fetch_daily.py` / `reauth.py` / `get_token*.py`（YouTube売上の取得・トークン更新。★最重要）
- `~/agent-skills/neta-forge/scripts/` … `scout_competitors.py` / `neta_slate.py`
- `~/agent-skills/neta-research/scripts/` … `fetch_channel_stats.py` / `verify_predictions.py` / `audit_thread_availability.py` / `score_stock_sheet.py`

### いま入ってるライブラリ（3.9.6のユーザー領域）

場所: `~/Library/Python/3.9/lib/python/site-packages`

主なもの（バージョン）:
- google-api-python-client 2.197.0 / google-auth 2.50.0 / google-auth-oauthlib 1.3.1
- google-auth-httplib2 0.3.1 / httplib2 0.31.2 / oauthlib 3.3.1 / requests-oauthlib 2.0.0

- pillow 11.3.0 / numpy 2.0.2 / pandas 2.3.3 / protobuf 6.33.6
- （bs4 / beautifulsoup4 もスキルで使用）

新しいPythonに移ると、**これらは入れ直し（pip install）が必要**。コードはほぼそのまま動くはず（下のテスト結果参照）。

## 4. 推奨するターゲット版

### 推奨: **Python 3.12 系（Homebrewで導入）**

理由（やさしく）:
- 3.12は今いちばん「枯れてて安定」してる本命。Google系・pandas・numpy・pillow 全部が公式対応済みで、トラブルが少ない。
- Homebrewで入れると、これ以降のアップデートも `brew upgrade` で楽。鍵やトークンには一切触らない。
- 3.13でもいいけど、ごく一部のライブラリで対応が遅れることがあるので、安定重視なら3.12が安全。

### 導入方法は2案。おすすめは A。

**A案: Homebrew（おすすめ）**
- `brew install python@3.12` で `/opt/homebrew/bin/python3.12` が入る。
- シンプルで分かりやすい。今後の管理も `brew` 一本。

**B案: pyenv（複数バージョンを切り替えたい人向け）**
- バージョンを何個も並べて切り替えられる。けど祐馬さんの用途（決まった版で安定運用）だと、ちょっと過剰。
- なので今回は **A案を推奨**。

> どちらも「システムの `/usr/bin/python3`（3.9.6）はそのまま残す」方式。古い方を消したりしないから、最悪いつでも元に戻せる。

## 5. 切替戦略（ここが本番。でも今日はやらない）

考え方: **古いPythonは消さず、新しいPythonを"足す"。スクリプトの向き先を1ジョブずつ少しずつ変える。一気にやらない。**

### ステップ0: 準備（壊さないための土台）
1. Homebrewを入れる → `brew install python@3.12`
2. 新Python専用の置き場（venv）を1個つくる。例: `~/.venvs/agent`
   - `/opt/homebrew/bin/python3.12 -m venv ~/.venvs/agent`
   - この中に、今のライブラリを入れ直す（pip install。下のリスト参照）。**システムは汚さない＝venvの中だけに入る。**
3. これで「新しい環境一式」が `~/.venvs/agent/bin/python` として用意できる。

### ステップ1: 一番リスクの低いジョブで試す（カナリア）
- まずはお金・認証に関係ない軽いジョブ1つ（例: `weekly_stocktake`）だけ、呼び出し先を新Python（venv）に向ける。
- 数日〜1週間ふつうに動くのを確認。

### ステップ2: Google認証ジョブを慎重に移す（★本丸）
- `youtube-revenue`（fetch_daily.py）を新Pythonに向ける。
- **トークン・鍵ファイルはそのまま使う（移さない・再取得しない）**。Pythonとライブラリだけ新しくなるイメージ。
- 1回手動実行して、YouTube売上がちゃんと取れる＝認証が通ることを確認してから自動運用に戻す。

### ステップ3: 残りを順に
- freee監視 → その他のシェル経由ジョブ、と1つずつ。各ジョブ、移したら必ず1回手動で動かして確認。

### 向き先の変え方（具体的にどこを書き換えるか）
- **launchd直叩きの3ジョブ**: plistの `<string>/usr/bin/python3</string>` を `<string>/Users/kojinn/.venvs/agent/bin/python</string>` に書き換え → `launchctl unload`→`load` で反映。
- **シェルスクリプト経由**: 各 .sh の中の `python3` / `/usr/bin/python3` を venv のpythonに向ける。
  - おすすめは「各スクリプトの頭で `PY=~/.venvs/agent/bin/python` と定義して、本文は `$PY` を使う」形に統一すると、次回からの切替が1行で済む。
- **.pyの `#!/usr/bin/env python3`**: 基本そのままでOK（呼び出し側でvenvのpythonを指定すれば、shebangは使われない）。直接 `./script.py` で実行する運用がある場合だけ要調整。

## 6. ロールバック手順（戻し方）

各ステップは独立してるので、おかしくなったジョブだけ元に戻せる。

- **launchd直叩き**: plistのpythonパスを `/usr/bin/python3` に戻して `unload`→`load`。それだけで3.9.6運用に復帰。
- **シェル経由**: `PY=` の行を `PY=/usr/bin/python3` に戻すだけ。
- 念のため、書き換える前のplist/シェルは `.bak-YYYYMMDD` でコピーを取ってから触る（このVaultの既存ジョブも .bak 運用してるので同じやり方）。
- 古い3.9.6・古いライブラリは消さないので、**「全部いっぺんに壊れて戻せない」が起きない**設計。
