## 4. 推奨するターゲット版

### 推奨: **Python 3.12 系（Homebrewで導入）**

理由（やさしく）:
- 3.12は今いちばん「枯れてて安定」してる本命。Google系・pandas・numpy・pillow 全部が公式対応済みで、トラブルが少ない。

- 3.13でもいいけど、ごく一部のライブラリで対応が遅れることがあるので、安定重視なら3.12が安全。

### 導入方法は2案。おすすめは A。

**A案: Homebrew（おすすめ）**
- `brew install python@3.12` で `/opt/homebrew/bin/python3.12` が入る。

- バージョンを何個も並べて切り替えられる。けど祐馬さんの用途（決まった版で安定運用）だと、ちょっと過剰。
- なので今回は **A案を推奨**。

> どちらも「システムの `/usr/bin/python3`（3.9.6）はそのまま残す」方式。古い方を消したりしないから、最悪いつでも元に戻せる。

## 5. 切替戦略（ここが本番。でも今日はやらない）

考え方: **古いPythonは消さず、新しいPythonを"足す"。スクリプトの向き先を1ジョブずつ少しずつ変える。一気にやらない。**

### 各 .sh の頭で `PY=` を一元定義する方式（全シェルに適用）

いまの .sh は `python3`（PATH頼み）と `/usr/bin/python3`（フルパス）がバラバラ。これだと**launchdで動いたときと、手で動かしたときでPythonが食い違う**事故が起きる（launchdのPATHは手動シェルと違うため）。

→ そこで、**各 .sh の冒頭に1行 `PY="/Users/kojinn/.venvs/agent/bin/python"` を置いて、本文の `python3` / `/usr/bin/python3` を全部 `"$PY"` に統一する**。

メリット:
- launchdでも手動でも、必ず同じPythonが使われる（食い違いゼロ）。
- 次回の切替やロールバックが、その1行を書き換えるだけで済む。
- どの .sh がどのPythonを使ってるか、頭の1行を見れば一目で分かる。

### ステップ0: 準備（壊さないための土台）

1. Homebrewを入れる → `brew install python@3.12`
2. 新Python専用の置き場（venv）を1個つくる。例: `~/.venvs/agent`
   - `/opt/homebrew/bin/python3.12 -m venv ~/.venvs/agent`
3. **いまの版を固める**: 3.9.6側で `python3 -m pip freeze > ~/.venvs/requirements.lock`（第7章）
4. **新venvへ同じ版で再現**: `~/.venvs/agent/bin/pip install -r ~/.venvs/requirements.lock`
   - これで「新しい環境一式」が `~/.venvs/agent/bin/python` として用意できる。**システムは汚さない＝venvの中だけに入る。**
5. **★FDA付与（祐馬さんの手作業・第8章）**。これをやらないとDriveが止まる。

### ステップ1: 一番リスクの低いジョブで試す（カナリア＝純計算）

- まずは**お金・認証・Driveに触らない純計算のジョブ**でテスト。
  - 例: `predict_score.py` 単体実行、または `python -m py_compile` 一式。
- これは「新Pythonでコードが動くか」だけを見る、一番安全な入口。

### ステップ2: FDA軽め（Drive読み書きはするが、お金・認証は無いもの）

- 例: `weekly_stocktake`（launchd直叩きの中で一番軽い）を新venvに向ける。
- **このときFDAカナリア（第8章）でDrive読み書きがEPERMにならないことを必ず確認**。

### ステップ3: お金・認証の重いジョブ（★本丸・最後）

- `youtube-revenue`（fetch_daily.py＝Google認証）→ `freee-uncleared-monitor`（お金）の順。
- **トークン・鍵ファイルはそのまま使う（移さない・再取得しない）**。Pythonとライブラリだけ新しくなるイメージ。
- 1回手動実行して、YouTube売上が取れる＝認証が通ることを確認してから自動運用に戻す。
- freee系は**実装者≠確認者**の検証ゲートを守る（AIが流して、祐馬さんが二重計上ゼロを目視）。

### 向き先の変え方（具体的にどこを書き換えるか）

- **launchd直叩きの3ジョブ**: plistの `<string>/usr/bin/python3</string>` を `<string>/Users/kojinn/.venvs/agent/bin/python</string>` に書き換え。
  - 反映は `bootout`→`bootstrap`（第11章で1ジョブだけ先に実証してから横展開）。
- **シェルスクリプト経由**: 各 .sh の頭に `PY=` を置いて本文を `"$PY"` に統一（上記方式）。
- **.pyの `#!/usr/bin/env python3`**: 基本そのままでOK（呼び出し側でvenvのpythonを指定すれば、shebangは使われない）。直接 `./script.py` で実行する運用がある場合だけ要調整。

## 6. ロールバック手順（戻し方）

各ステップは独立してるので、おかしくなったジョブだけ元に戻せる。

- **launchd直叩き**: plistのpythonパスを `/usr/bin/python3` に戻して `bootout`→`bootstrap`。それだけで3.9.6運用に復帰。

## 7. ライブラリの版固定（pip freeze → 同版再現 → 必要分だけ更新）

版をバラバラに入れ直すと「前は動いてたのに新版で挙動が変わった」事故が起きやすい。なので順番を守る:

1. **いまの版を固める**（3.9.6側で）:
   ```
   /usr/bin/python3 -m pip freeze > ~/.venvs/requirements.lock
   ```
2. **新venvへ同じ版で再現**:
   ```
   ~/.venvs/agent/bin/pip install -r ~/.venvs/requirements.lock
   ```
   - もし3.9時代の版が3.12で入らないものがあれば、その1個だけ最新へ。記録を残す。
3. **必要な分だけ上げる**（Google系のセキュリティ更新など）。一気に全部上げない。
4. lockファイルはVaultに控える（`06_エージェント運用/` 配下に日付つきで）。

メインで使うライブラリ（lockに含まれるはず）:
```
google-api-python-client google-auth google-auth-oauthlib google-auth-httplib2 \
oauthlib requests-oauthlib requests urllib3 certifi httplib2 \
pillow numpy pandas protobuf beautifulsoup4
```

## 8. ★FDA（フルディスクアクセス）の手当て ＝ 一番大事な見落とし

### なぜ必要なの？（やさしく）

macOSは「どのアプリ／実体ファイルがDriveなどの保護領域を読めるか」を**実体ファイル単位**で許可してる。
いまは `/bin/bash` と `/usr/bin/python3` がFDAを持っていて、その文脈で動く子プロセス（python）もDriveを読めてる。
これは `run_kicho.sh` のコメントにもはっきり書いてあるよ:

> `# 1) 直接実行（/bin/bashはFDA持ち。子のpython3もこの文脈ならDriveを読める）`

**ここで `/usr/bin/python3` を venv の新しい python（`~/.venvs/agent/bin/python`）に差し替えると、新しい実体にはFDAが付いてない**。
→ 経費精算・売上証憑・2nd-Brain への読み書きが **EPERM（権限なし）で止まる**。これはVaultのEPERM恒久対策の根幹にぶつかる、一番こわい落とし穴。

（補足: いまこのプラン作成中に `TCC.db` を読もうとしたら "authorization denied" になった。これはFDAが実体単位で効いてる証拠でもあるよ。）

### 祐馬さんの手作業（必須・管理者操作）

切替の**前**に、新しいpythonの実体にFDAを付ける:

1. `~/.venvs/agent/bin/python` の**実体パス**を確認する（symlinkなので実体を辿る）:
   - `readlink -f ~/.venvs/agent/bin/python`（→ `/opt/homebrew/.../python3.12` などの実体が出る）
2. **システム設定 → プライバシーとセキュリティ → フルディスクアクセス** を開く。
3. 左下の鍵を外して（管理者パスワード）、`＋` から上で確認した**実体のpython**を追加してONにする。
   - Finderで隠しフォルダへ行くときは `Cmd+Shift+G` でパスを直接貼ると早い。
4. 念のため、`/bin/bash`（既にFDA持ちのはず）の状態も確認しておく。

### 付与した当日のカナリア確認（Drive読み書き）

FDAを付けたら、**お金・認証は触らない**まま、Driveの読み書きだけ確認する:

- 読み: `~/.venvs/agent/bin/python -c "import os; print(os.listdir(os.path.expanduser('~/Library/CloudStorage/GoogleDrive-corocoro.business@gmail.com/マイドライブ/経費精算')))"`
  - → 中身が一覧できればFDA読みOK。
- 書き: 同じ場所に一時ファイルを1個作って→消す（`canary_YYYYMMDD.txt` などを書いて即削除。証憑には一切触らない）。
  - → EPERMが出なければFDA書きOK。

ここがEPERMで止まるようなら、**先に進まずFDA設定を見直す**（横展開しない）。

## 11. launchd反映方式の実証（bootout/bootstrap を1ジョブで先に確認）

plist書き換え後の反映は、macOSの版によって `unload/load`（旧）と `bootout/bootstrap`（新）で挙動が違うことがある。
横展開の前に、**一番安全な1ジョブ（weekly-stocktake など）でだけ**、この手順が通るか実証する:

1. plistを `.bak-YYYYMMDD` でバックアップ
2. `launchctl bootout gui/$(id -u)/com.claude.weekly-stocktake`
3. plistのpythonパスを `$PYBIN` に書き換え
4. `launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.claude.weekly-stocktake.plist`
5. `launchctl kickstart -k gui/$(id -u)/com.claude.weekly-stocktake` で1回手動起動
6. ログで成功を確認 → **ここが通って初めて**、他のジョブにも同じ手順を横展開

> 反映でコケたら即 `.bak` に戻して `bootstrap`（第6章ロールバック）。
