# Codexメイン移行 設計レビュー

作成日: 2026-07-01
対象: `00_システム/20_Agent_Portable/specs/codex-main-vendor-neutral-migration.md`

## 結論

この設計は、現状のベストプラクティスとして妥当。

ただし、完成状態ではない。今の正しいゴールは「Claudeを一気に外す」ではなく、Codexを日常メイン入口にし、既存Claude依存ジョブを `agent-run` 経由で1件ずつ安全に移すこと。

## 5視点レビュー

### 1. アーキテクト視点

判定: PASS

理由:
- AIごとにスクリプトを分岐させず、`agent-run` をvendor seamにする方針が正しい
- Obsidianを正本、`agent-skills` をスキル正本、`agent-adapters` を入口に分離できている
- Hermes/OpenClawを即本線化せず、adapter候補として扱うのが安全

注意:
- `agent-run` のvendor分岐が肥大化したら、共通オプション変換とvendor固有実行を分ける

### 2. SRE / 自動化運用視点

判定: PASS

理由:
- launchd停止・本番差し替えを人間承認にしている
- dry-run、read-only、構造化出力、非更新確認、Watchtower更新をゲートにしている
- `com.claude.*` のラベル名変更を急がない判断が正しい

注意:
- 定期実行を切り替える時は、最低1周期のshadow確認を残す
- rollback手順がないジョブは切り替えない

### 3. セキュリティ / プライバシー視点

判定: PASS

理由:
- 外部送信・本番データ・認証・個人情報を `あり` と仮定している
- 秘密情報、Cookie、セッション、ログ生データをSecond Brainへ保存しない
- 外部エージェントへ渡す情報を匿名化仕様・エラー要旨に限定している

注意:
- Gmail削除、会計書込、Drive証憑、Discord投稿は別案件として扱う

### 4. プロダクト / 日常運用視点

判定: PASS

理由:
- 祐馬さんの日常入口をCodexへ寄せる判断が明確
- Channels/Discordを本線から外し、受付・通知・過去ログ参照に限定している
- 「どんなAIでも使える仕組み」に寄せるため、特定AIへの再ロックインを避けている

注意:
- スマホ受付を完全に止めると日常運用が不便になる可能性があるため、代替InboxができるまでChannels/Discordは残す

### 5. QA / 完成判定視点

判定: PASS WITH CONDITIONS

理由:
- 完成条件が入口、継ぎ目、棚卸し、移行、脱ロックインに分解されている
- `thread-format-learning` を低リスクパイロットにしたのは妥当

残条件:
- 次の候補を1件だけ選ぶ
- dry-run wrapperを作る
- `read-only` Codex実行で外部投稿・永続書込なしを確認する
- 独立レビュー後に本番切替可否を判断する

## 最終判断

このフローは採用してよい。

禁止:
- Claude Channels停止を先にやる
- `~/.claude/scripts` を一括Codex化する
- launchdラベル名を一括変更する
- 会計/Gmail/削除/Discord投稿系を最初の移行対象にする

推奨:
- 次の移行候補は `script-learning`、`daily-knowledge-extract`、`knowledge-gardener` のどれか1件
- 迷う場合は、外部投稿と永続書込を止めやすい `script-learning` から始める
