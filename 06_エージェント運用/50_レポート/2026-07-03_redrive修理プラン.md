### Goal 4 / D2: 上限中にattemptsを燃やさない

推奨案:

`limit_redrive.py process` の冒頭で現在の上限状態を確認し、上限中なら処理を始めずreturnする。さらに個別処理中に上限系エラーが出た場合も、attemptsを増やさずqueueへ戻す。

変更ファイル:
- `~/.claude/scripts/limit_redrive.py`

関数:
- `probe()`
- `process()`
- 新規 `is_limit_output(out: str) -> bool`
- 可能なら `run_task()` 相当の小関数に分離

差分規模:
- 30〜70行程度

推奨設計:

1. `is_limit_output()` を追加し、次の文字列を上限系として見る
   - `weekly limit`
   - `session limit`
   - `usage limit`
   - `temporarily limiting`
   - `You've hit your weekly limit`
   - `resets`
2. `probe()` も `is_limit_output()` を使う
3. `process()` はqueue処理前に `probe()` を実行する
4. `probe()` が未回復なら、attemptsを増やさず `ledger.txt` に `process延期: 上限中` を記録してreturn
5. 個別実行で上限系エラーが出た場合は、`work` を `queue` へ戻し、attemptsを増やさずreturn
6. 上限以外の失敗だけattemptsを増やす

注意:
- 既存の `GUARD_AGE_H=2` は、上限中の古いqueueにも確認モードを発火させる。D2修正では、`probe()` が上限中と判定した場合は鮮度ガードより前に止める。上限が明けた後も古すぎるqueueなら確認モードへ降格してよい
- 無限リトライ防止は、attemptsではなく「上限中は処理しない」こととWatchtower滞留検知で担保する

### Goal 5 / D3: 確認モード降格が無通知で沈黙しない

推奨設計:

#### 5-1. 通知成否ログ

`notify()` を戻り値ありにする。

```python
def notify(msg: str, channel: str = MAIN_CHANNEL) -> bool:
    r = subprocess.run([...], capture_output=True, timeout=30)
    ok = r.returncode == 0
    log(f"notify {'OK' if ok else 'FAIL'} channel={channel}")
    return ok
```

確認モード降格時:
- `CONFIRM_FLAG.write_text(...)`
- 通知を試みる
- 通知OK/FAILをledgerへ必ず書く

通知文面案:

```text
📦 redrive確認モードに入ったよ。
預かり依頼が古くなったため自動配達を止めています。
処理するなら「預かり分やって」、不要なら「redrive残骸整理GO」と言ってください。
```

#### 5-2. Watchtower 24時間超検知

`check_redrive(now)` でflagのmtimeを読み、24時間超なら通常の「確認モード中」より強いWARNにする。

```python
flag = base / "confirm-mode.flag"
if flag.exists():
    age_h = (now.timestamp() - flag.stat().st_mtime) / 3600
    if age_h > 24:
        results.append(("warn", "redrive確認モード滞留", f"confirm-mode.flag が{age_h:.1f}時間残存。自動配達停止中"))
    else:
        results.append(("warn", "redrive配達", "確認モード中..."))
```

Watchtowerは既にWARNをDiscord通知する仕組みを持つため、新しい外部送信経路は増やさない。ただし警告項目の追加なので、危険操作承認が必要。
