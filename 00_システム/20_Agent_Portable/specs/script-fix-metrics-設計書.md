## 1. 背景と目的

台本の「修正→ルール化→機械化」ループはすでに実働している（7/3中国大返し検品→即日ルール化→ローテ系は `assemble_csv.py` へ機械化）。しかし：

1. **ルールが増えても、修正が減っているかを誰も測っていない**（1本あたり修正ラウンド数の記録なし）
2. **新ルールが機械ゲートに反映されたかの追跡がない**（proseのまま放置されても気づけない）
3. 「同じ指摘2回で昇格」の判定材料が制作ログ頼みで、機械的に数えられない

**目的**: 出荷1本ごとに「何回直したか・どのルールに違反したか」を1行記録し、月次で「ルールの効き」を答え合わせする。[[予測精度]]（出す前の予測を出した後の実績で検証）と同じ思想を、台本品質に適用する。

**ゴール状態**: 「ルールを足したのに同じ違反が減らない」「このルールはprose止まりで2回踏まれた＝機械化しろ」が**数字で**言える。

## 4. データ設計

### 4.1 ルールID体系（Phase 0）

- **R系**: データ確定ルール（既存 R1〜R4。今後 R5…）
- **F系**: 修正パターン（既存9項目に F01〜F09 を付与。今後 F10…）
- **ID再利用禁止**。ルール廃止時は打ち消し線＋`[廃止 YYYY-MM-DD]` で残す（git履歴が担保）
- **執行者タグ**（ルールを守らせている主体）:
  - `機械:assemble_csv` — スクリプトが機械適用（違反が構造上起きない）
  - `機械:qa` / `機械:sec48` — 機械検品でFAILになる
  - `LLM:step45` / `LLM:採点` — LLM監査・盲検採点で捕まえる
  - `prose` — 文章のみ＝執筆AIの注意力頼み（**最弱・機械化候補の母集団**）
- 記法（台本執筆ルール.md の各項目行頭に付与・grep可能な固定フォーマット）:
  `**[F05|機械:assemble_csv]** 話者ローテの飛びは禁止…`
- レジストリの正本は台本執筆ルール.md のインラインタグ。機械可読が必要な処理はこの固定記法をパースする（別JSONは作らない＝二重管理回避）

### 4.2 記録スキーマ（fix_metrics.jsonl・追記のみ・1行=1イベント）

**type=shipment（出荷時に1行・必須）**
```json
{"schema": 1, "type": "shipment", "date": "2026-07-14",
 "folder": "2026-07-03_中国大返し", "theme": "中国大返し",
 "rounds_ai_regen": 2, "rounds_owner_fix": 1,
 "gate_fails": [{"gate": "qa_hard", "detail": "字数", "count": 1},
                 {"gate": "step45", "detail": "講義臭", "count": 3}],
 "rule_violations": ["F05", "F07"],
 "new_rule_candidates": ["解説直後の下スタート禁止"],
 "note": ""}
```

**type=fix_event（出荷後に池田さんの修正指示が来たら都度1行）**
```json
{"schema": 1, "type": "fix_event", "date": "2026-07-16",
 "folder": "2026-07-03_中国大返し",
 "rule_violations": ["F08"], "severity": "minor",
 "note": "退場レス1本カット指示"}
```

- **カウントの定義**: `rounds_ai_regen`＝検品FAILによる再生成・修正サイクル数（同一FAIL K回打ち切りのKと同じ数え方）／`rounds_owner_fix`＝池田さんの指摘で直した回数（出荷前）。出荷後の指摘は fix_event で別記録
- `rule_violations` はID列。まだIDのない新種の指摘は `new_rule_candidates` に文字列で入れる（→月次で「2回出たらID化」判定の材料）
- 破損対策: 集計側は1行ずつJSONパースし、壊れた行はスキップして**警告に出す**（黙って捨てない）

## 5. コンポーネント詳細

### 5.1 record_fix_metrics.py（Phase 1・記録CLI）

- 使い方（出荷時・パイプラインSTEP7後に1コマンド）:
  `python3 record_fix_metrics.py ship --folder 2026-07-03_中国大返し --ai-regen 2 --owner-fix 1 --violations F05,F07 --gate-fail qa_hard:字数 --gate-fail step45:講義臭:3`
- 事後修正: `record_fix_metrics.py fix --folder ... --violations F08 --severity minor --note "..."`
- 挙動: バリデーション（ルールIDが台本執筆ルール.mdのタグに実在するか照合）→ JSONLへ追記 → 追記した行を表示して終了。**既存行の書き換え・削除はしない**
- 呼び出し規約: creative-thread-pipeline と youtube-pipeline の出荷手順に「記録1行」を追記（SKILL.md側は1行の追記のみ。ゲート化はしない＝まず習慣、詰まったらSTEP7の受入条件に昇格を検討）

### 5.2 fix_metrics_report.py（Phase 2・月次集計・read-only）

- 入力: fix_metrics.jsonl＋台本執筆ルール.mdのタグ＋video_ledger.json（出荷数の突合用）
- 出力: `台本ルール効き.md` を全再生成＋標準出力にサマリ。**ルールブック・台帳・JSONLへの書き込みは一切なし**
- レポート内容:
  1. 📉 **修正コスト推移**: 1本あたり総修正ラウンド（直近3本移動中央値）の月次推移
  2. 🔁 **ルール別違反ランキング**: ID・執行者タグ・追加日以降の違反回数
  3. ⚙️ **機械化候補**（提案のみ）: `prose`/`LLM`タグで違反2回以上のルール
  4. 🧟 **効いてないルール**（提案のみ）: ルール化後も同一違反が続くもの＝書き方が悪い/ゲート未反映
  5. 🆕 **ID化候補**: `new_rule_candidates` に2回以上出た文字列
  6. 🕳️ **記録漏れ**: 動画台帳の出荷数とshipment行数の差分（**欠測は自己申告**＝沈黙が正常の逆適用）
- 実行タイミング: 当面**オンデマンド**（月初のscript-learning報告を見たついでに手動1コマンド）。1〜2ヶ月運用して安定したら、script-learningへの統合レッグを**別途提案**（shadow判定2026-08-02の後。勝手に統合しない）

### 5.3 昇格・剪定の運用（Phase 3・人間ループ）

- 月次レポートの「機械化候補」を見て、採用するものだけ Codex実行ブリーフ化（§4.8チェック項目の追加 or assemble_csv側の適用）
- 機械化したら台本執筆ルール.mdの執行者タグを更新（`prose`→`機械:sec48`）＝**タグ更新までが機械化の完了定義**
- 3ヶ月違反ゼロの`prose`ルールは「定着」として短文化（読込負荷の剪定・池田さん承認制）
- 自動書き換えは**永久にしない**（提案→承認→人が反映。実装者≠確認者ゲート維持）

## 6. 役割分担（LLM判断 と script機械適用）

## 7. KPI（この仕組み自体の答え合わせ）

- **北極星**: 1本あたり総修正ラウンド数（ai_regen＋owner_fix、3本移動中央値）が**下がり続ける**こと
- 補助1: ルール化済み違反の再発件数（理想＝0。再発＝ルールが効いてない証拠）
- 補助2: `prose`タグのルール数（機械化バックログ。減るほど体質が強い）
- この仕組み自体が3ヶ月使われなかったら（記録漏れ>50%）、設計が重すぎる証拠＝軽量化 or 廃止を判断する（ゾンビ化させない）

## 8. エッジケース・失敗モード

| ケース | 挙動 |
|---|---|
| 記録忘れ | 月次レポートが台帳と突合して「🕳️記録漏れn本」を必ず表示（黙らない） |
| ルールID打ち間違い | record CLIがタグ照合でエラー→追記しない（fail-close） |
| JSONL行破損 | 集計は壊れた行をスキップ＋警告表示（黙って捨てない） |
| ルール廃止 | ID再利用禁止・打ち消し線で残す。集計は廃止日以降の違反を「廃止済違反」として別枠表示 |
| Projects/youtube が未pushでMac故障 | satellite-autocommit（10分毎）の対象内＝既存バックアップ層でカバー |
| 数字とルールブックの矛盾 | 数字の正本はJSONL、ルールの正本はルールブック。矛盾時はレポートに両論併記して人が判断 |
