# 設計書：creative-thread-pipeline（創作スレ台本 一気通貫スキル）

- 作成: 2026-06-20
- 起点: 平将門の祟り回の制作過程（創作スレ→YMM4台本）を再現可能なスキルに昇華する
- 状態: **設計確定（平将門の全工程と照合しレビュー反映済 → §13・実装待ち）**

---

## 1. 背景と目的
平将門回で「創作スレ（オリジナル2chスレ）→ 解説者パート → YMM4取り込み用CSV/xlsx」を作ったが、
**組み立て(アセンブル)とxlsx化・ビューアは使い捨てスクリプト(gen_csv.py / make_xlsx.py)で手組み**だった。
この工程を**毎回再現できるスキル**にする。生成系は既存スキルを呼び、本スキルは
**「組み立て規約 ＋ 検品ゲート ＋ 出荷物の生成」**を担う。

## 2. 既存スキルとの関係（重複回避が最重要）
| スキル | 役割 | 本スキルとの関係 |
|---|---|---|
| `youtube-pipeline` | **実スレscrape版**の6ステップ一気通貫（thread_scraper.pyで本物スレ取得） | 兄弟。creative版が本スキル。結合エンジンは共有化を狙う |
| `creative-thread-gen` | 創作スレ（前半/後半）生成＋qa_check（16+ゲート） | **STEP2で呼ぶ**（×2本）|
| `youtube-script-parts` | 冒頭・解説・前置き・締め生成 | **STEP3で呼ぶ**（解説は対象用語ぶん）|
| `youtube-script-checker` | 規約3段階チェック | **STEP5で呼ぶ** |
| `ymm4-dic-generator` | 読み仮名dic（毎回書き換え・漏れ/誤読ゼロ） | **STEP6で呼ぶ** |
| `ymm4-pronunciation-checker` | YMM4取り込み後の発音照合 | 任意・STEP7（dic適用後）|

→ **本スキルは生成を再実装しない。オーケストレーション＋アセンブル＋出荷に徹する。**
→ アセンブル規約は `scripts/assemble_csv.py` ＋ `refs/組み立て規約.md` に集約し、
   将来 youtube-pipeline もこの共有エンジンを採用して規約の二重化を防ぐ。

## 3. スキル定義
- **名前**: `creative-thread-pipeline`
- **トリガー**: 「創作スレ台本」「オリジナルスレで台本」「スレ台本一気通貫」「転載なしで台本」「creative pipeline」
- **立ち位置**: ネタ確定後〜YMM4取り込み直前までの創作スレ版フルパイプライン
- **各ステップ後にユーザー確認**（youtube-pipeline踏襲・暴走防止）

## 4. パイプライン全体
```
STEP1 ネタ確定（テーマ＋前半=タイトルネタ/後半の振り分け）
      ※ネタ出しは neta-research / thread-research-planner（任意・前段）
STEP2 創作スレ生成 ×2  →  /creative-thread-gen（前半・後半）
      ・品質バー（→ §4.5）：ハードゲート【PASS必須】／WARN系AI癖【非ブロック＝出荷可】／
        フル本物化(選択肢C)【任意STEP】。「PASSまで無限ループ」を避ける＝Cは出したい回だけ明示実行。
      ・出力＝「番号: 本文」形式（>>Nアンカー保持・欠番の不規則化は creative-thread-gen 側の責務）
STEP3 解説対象＆挿入位置の選定 → 解説者パート生成  →  /youtube-script-parts
      ・難用語/固有名から解説対象をN個抽出 →【提案→確認】
      ・各解説の挿入位置も【提案→確認】＝「対象用語クラスタが一段落した直後（次の話題へ移る前）」
        ※「用語初出の直後」ではない（初出は壁の途中になりがち）。平将門例：大蔵省は8初出だが解説は15直後
      ・冒頭/前置き(1本)/締め＋解説×N を生成
STEP4 アセンブル【本スキルの心臓部】→ /scripts/assemble_csv.py
      ・構成順に結合し、話者・大/下・返信先・解説挿入・句読点改行を機械付与（§6）
      ・スレは read-only（アセンブルは番号を振り直さない＝§7）
      ・出力＝CSV(utf-8-sig) ＋ xlsx ＋ ~/Desktop複製（会話ビューアは作らない）
STEP4.5 敵対的"本物らしさ"監査【必須・機械ゲートの穴を埋める】→ §4.6 の5レンズLLM監査
      ・must_fix（講義臭=ナレVO化／二重主語/文末表現の文頭/会話流れ）を捕捉→修正→再qa →【提案→確認】
STEP5 規約3段階チェック → /youtube-script-checker（8層＋LLM文脈＋ファクト）
      ・近代政治色（GHQ/米軍/進駐軍 等）検出→【歴史一般化方針】で修正（→ §4.7）
      ・現代の実在個人はカテゴリ7で🔴→匿名化
STEP6 読み仮名dic → /ymm4-dic-generator（毎回書き換え・漏れ/誤読ゼロ）
STEP7 (任意) 発音チェック /ymm4-pronunciation-checker、タイトル/サムネ/概要欄 /youtube-script-parts、投稿予約
```

## 4.5 品質バー（3段の合格ライン）
- **ハードゲート＝PASS必須**：字数／禁止語(死/殺す/アホ/バカ/クソ/カス 等)／SLUR・ヘイト／
  規約🔴／時代タグFAIL／ASCII"。**1つでも引っかかれば出荷不可**（自動承認しない）。
- **WARN系AI癖＝非ブロック（出荷可）**：SCAF密度／感情温度／壁化／9f密度／単一語尾過集中／
  ナレ接着頭／長文独り語り 等。デフォルトはWARNのまま出荷可（毎回フル平準化はしない）。
- **フル本物化（選択肢C）＝任意STEP**：長文を淡々・ぶつ切りに総リライトしWARNも全消し。
  看板回など"本物2ch質感"を最優先する時だけ実行。重いので毎回はやらない。
- 「本物らしさの天井」（評価締めの同一ビート反復／時系列一本道／死にレス少なめ）は**意図的に追わない**。

## 4.6 敵対的"本物らしさ"監査（STEP4.5の中身）※機械ゲートが拾えない content を捕捉
ultracode 5レンズ（並列LLM監査）。平将門で must_fix を出した実績ある工程＝**これが無いと品質が落ちる**。
- レンズ＝①なんJ語感 ②アンカー整合 ③講義臭 ④AI臭残留 ⑤規約
- 必ず捕捉する content パターン（機械ゲート化しにくい＝今回の must_fix）：
  - **講義臭＝ナレVO化**：元号＋西暦キャプション＋終止形現在で長文を始める（掲示板の実況ナレ化）
    → 掲示板口調へ（例「940年、天慶3年やな。…敗れてまう」）。
  - **二重主語の文法よれ**（「将門は…討伐軍が苦戦しとった」）→ 述語一本化。
  - **文末専用表現の文頭使用**（「知らんけど」を文頭に置く 等）→ 文末へ。
  - **返信アンカーが会話の流れと噛み合うか**（§6.6 の後方ジャンプ＝①66）。
- 偽陽性対策：サイズ交絡・辞書バイアス・スクレイプ崩れは棄却（平将門の8軸監査で実証済）。
- must_fix を出したら該当レスを修正→**再qaでPASS維持を確認**→ユーザーに差分提示→確認。

## 4.7 近代政治色の歴史一般化方針（STEP5の修正指針）
youtube-script-checker が近代政治の固有名（GHQ／米軍／進駐軍／特定政党 等）を検出したら、
**実在の近代政治色を排し歴史的記述へ一般化**して修正する：
- 例：GHQ/進駐軍 → 「戦後の区画整理」「重機」「国家事業」「お役所」。
- 検出だけで終わらせず、この一般化までを修正方針として明記（平将門で実施済）。

## 4.8 最終受入チェック（出荷前の決定的検証）※今回手作業だった検証をスキル化
組み立て後、**機械的に整合性を検証**してから「完成」と言う（平将門で手打ちした検証の常設化）：
- 解説が指定本数ぶん正しい位置に挿入されているか（スレ①N本/スレ②N本）
- **大に `>>` が付いていないか**（下大→大の帰結）／大の直後に下・>>が無いか
- 返信先 `>>M` が全て**自分より前**を指すか（未来/自己参照ゼロ）
- スレタイが各スレ先頭・男性Aか
- CSV：列数・utf-8-sig・句読点改行・行数
- 規約：削除対象語（現代実在個人の実名 等）が本文に残っていないか（例 太田=0）
- 不一致が1つでもあれば**完成と言わず**該当を修正→再検証。

**出力（保存先 = `01_プロジェクト/YouTube/創作スレ下書き/YYYY-MM-DD_テーマ`）**：

## 6. CSV組み立て規約【正本＝refs/組み立て規約.md にも複製】
### 6.1 列構成
`登場人物, セリフ, 備考, 番号`（番号は「連番」または「連番>>返信先連番」）
### 6.2 構成順（備考列のセクション）
冒頭 → スレ① →〔解説:用語A〕→〔解説:用語B〕→ 前置き → スレ② →〔解説:用語C〕→〔解説:用語D〕→ 締め
※解説は**対象用語クラスタが一段落した直後（次の話題へ移る前）**にスレの合間へ挿入。
　**「用語初出の直後」ではない**（初出は壁の途中になりがち）→ 位置は §4 STEP3 で提案→確認。
### 6.3 話者
- スレタイ（各スレ1レス目）＝**男性A**
- 以後 **男性A→女性B→男性C→女性D→男性E** をレス順にローテ
- 解説者パートは全部「解説者」
### 6.4 大（長文ラベル）
- `>>`/`↑`除去後の本文が **88字以上 → 大**
### 6.5 下・返信先（番号列 `N>>M`）
- 返信（本文が`>>`始まり or スレタイ反応）→ **下** ＋ 番号 `N>>M`（M=返信先のスレ内連番）
- **スレタイへの反応＝`>>1`**（各スレ冒頭の反応バースト）
- **下大→大（大優先）**：長文かつ返信でも「大」にし、下も>>も付けない
- **大の直後のレスは下も>>も付けない**（返信でも素の番号）
### 6.6 返信先は「会話の流れ」準拠（最重要・平将門①66の教訓）
- 返信先は**直近 or 同一話題のwall**へ。
- **話題が変わった後ろのレスへ戻るジャンプは禁止**（間に別話題が挟まる>>は不自然）。
- 自動導出時、後方ジャンプ（gap≥3かつ間に別話題）を検出したら**警告して直近へ寄せる**。
### 6.7 句読点改行（TTS用）
- セリフは `。、！？` の直後で改行（既存改行は一旦解除してから付与）
### 6.8 文字コード
- CSV=UTF-8(BOM付き/utf-8-sig)・LF。xlsxはセリフ列を折返し表示。
### 6.9 句点(。)・読点(、)は【あえてゲートしない】※将来の事故防止メモ
- 本物2chは句点が極端に少ない＝AI臭の"穴"ではあるが、**うちのCSVはTTS用に句読点改行を使う**ため
  句点/読点を削ると改行ロジックが壊れる。よって**句点系の整然さは意図的にゲート対象外**にしている。
- 将来この"穴"を機械ゲートで塞ごうとしないこと（TTSフローと衝突する＝平将門8軸監査での確定判断）。

## 7. 役割分担：LLM判断（提案→確認）と script機械適用 ※ドライランで確定
**ドライラン知見(2026-06-20)**：返信マップのregex自動導出を thread2 で検証→手マップと**50%しか一致せず**
（完全一致20/40・先違い6・取りこぼし14・余計13、会話流れWARNも誤爆17）。
原因＝本物スレの`>>`は疎ら（qa-9g対策で壁にだけ付く）で、「誰が誰に返してるか」は本文を読まないと分からない。
→ **返信マップ・解説位置・解説対象は決定的に導出できない＝LLM判断タスク**と判明。役割を分ける：

### 7.1 LLM判断（スキルが読んで推論→提案→ユーザー確認→manifestへ）※regex化しない
- **返信マップ（会話の返信グラフ）**：スレを読み、誰が誰に返してるかを §6.5/6.6 のルールで解釈。
  スレの明示`>>N`は手がかりの一部に過ぎない。短い反応→直近wall／スレタイ反応→>>1／後方ジャンプ(§6.6)回避、
  を**LLMが判断**して決める（regexは50%で破綻＝実証済）。
- **解説の挿入位置**：対象用語クラスタの末尾（§6.2）を読んで判断。**初出直後ではない**。
- **解説対象の選定**：難用語/固有名トップN。
- いずれも **提案→確認ゲート必須**。確定値を §8 のmanifestに載せて 7.2 へ渡す。

### 7.2 script機械適用（assemble_csv.py が決定的に処理＝手判定ゼロ）
- 大/下ラベル付与：88字+=大／返信=下／**下大→大**／**大の直後は下も>>もなし**（§6.3-6.5）。
- 番号列：スレ内連番を新規付与＋（LLMが決めた）返信マップを `N>>M` で反映。
- 句読点改行（§6.7）、解説の**指定位置**挿入、CSV(utf-8-sig)/xlsx/MD出力（§6.8）。
- **スレは read-only（番号を振り直さない）**：内部`>>N`→連番はマッピングするだけ。
  ※平将門で「post-30一括振り直しで本文>>が壊れた」のは**スレ側を改変したから**＝二度とやらない。
  欠番の不規則化（AI臭回避）は **creative-thread-gen 側の責務**（qaが欠番率/不規則性を見る）。

## 7.5 自動化グラデーション（確認ゲート → 学習 → 段階自動化）※設計の軸
オーナー方針＝**「初期は各STEPで確認、ナレッジ/学習が溜まったらフル自動化も視野」**。
- **初期＝全STEPで確認ゲートON**。各STEP完了時に結果を提示し、ユーザー承認を取ってから次へ。
- 各STEPの判断を `learning-log.md` に記録。ただし**"値"でなく"判断ルール/修正パターン"を記録**する
  （毎回ネタが違うので具体値は汎化しない）。例：「解説対象＝難読固有名トップN」「①66型の後方ジャンプは直近へ」。
- **成熟度トラッカー**：同種の判断がK回連続（暫定K=3〜5）で「そのまま承認」されたSTEPは、
  スキルが「このSTEP、自動化していい？」と**提案**する（**提案専用＝勝手に自動化しない**）。
- ユーザー承認で**そのSTEPだけ自動モードにflip**（他STEPは確認のまま個別運用）。
- 自動化の進む順（判断が安定しやすい順）：STEP4アセンブル＞STEP6 dic＞STEP3解説対象選定＞…。
- creative-thread-genの学習ループ（収集→学習→**承認**→検証付き適用）と同一哲学＝
  **金/出荷に関わる所は実装者≠確認者の検証ゲートを残す**（rules.md準拠）。
- 完全自動運用に移っても、STEP5規約チェックの🔴とqa_checkのFAILは**常にハードゲート**（自動承認しない）。

## 8. 同梱物（skill配下）
- `scripts/assemble_csv.py` … threads(2)＋parts → CSV/MD（gen_csv.pyの汎用版）

    `{出力base, 前置きパス, 締めパス, 冒頭パス,

- `refs/組み立て規約.md` … 6章の正本（人間レビュー用）
- `refs/サンプル/` … 平将門の最終CSVを参照見本として同梱

- ※会話ビューアのテンプレは同梱しない（不要との確定）

## 9. エッジケース・失敗モード
- 解説対象がスレに出てこない → ユーザーに代替提案（用語追加 or 解説スキップ）
- qa_checkがPASSしない → creative-thread-genに差し戻し（本スキルは検品で止める）
- 大が連続しすぎ/壁化WARN → スレ側の問題なので生成スキルへ戻す（アセンブルでは直さない）
- 返信先の自動導出が会話流れ警告を出す → ユーザーに該当番号を提示して確認
