## 0. 一言でいうと

**「Fableが考えて、Codexが手を動かして、祐馬さんが境界を越える許可を出す」**。
AI運用の再構築案件（修理・整備・改善）専用の型。日常業務（YouTube・売上・レシート等）は今のまま何も変えない。

## 1. 変えないこと（現行運用の保護）

- 正本は Obsidian（ローカル）／会計はfreee／証憑はDrive
- 日常の入口はCodex（移行フェーズ1）のまま。Codex切替完了とは扱わない
- 既存の自動化・launchd・監視・Discord受付はこのルール自体では何も動かさない
- 実装者≠確認者の検証ゲートは維持

変えるのは1つだけ: **「AI運用の再構築案件」の進め方に、Fable判断→Codex実行→Fable検証 の型を追加する**。

※注記: 移行設計書（codex-main-vendor-neutral-migration.md §8）の標準編成では Commander=Codex。本案では**再構築案件に限り Commander=Fable** とする。採用時に§8へ注記1行を入れて整合を取る（これが唯一の既存文書との衝突点）。

## 2. 役割（3行で）

| 役割 | 担当 | やること |
|---|---|---|
| 判断（司令塔） | Fable | 調査・原因特定・優先順位・ブリーフ作成・結果の独立検証 |
| 実行 | Codex | ブリーフの範囲内だけ実装・修正・記録 |
| 承認 | 祐馬さん | ブリーフをCodexへ渡す行為＝実行承認。境界越えのGO |

FableはCodexを直接起動できない。**祐馬さんがブリーフを貼るその手が、承認ゲートそのもの**（仕組みで人間承認が必ず挟まる）。

## 3. Fableが判断すること

1. 現状調査（読み取りのみ）と原因の特定
2. 直す順番の決定（危険／重要／後でよい）
3. Codexへのブリーフ作成——**目的・触ってよい場所・やらないでほしいこと・完了条件・記録先** を必ず全部入れる
4. Codex作業後の独立検証（実装者≠確認者。OKなら次へ、NGなら差し戻しブリーフ）
5. 祐馬さんに聞くべきことの選別（本当に迷ったら聞く。過剰確認はしない）

Fable自身が手を動かしてよい範囲: 読み取り調査／ブリーフ・レポート作成／task_cli登録／自分のメモリ更新。**実装はしない**。

## 4. Codexが実行すること

1. ブリーフに書かれた範囲の実装・文書修正・検証
2. 作業ログへの記録（毎回・省略禁止）
3. ブリーフ外の発見は**実行せず**報告に載せる（勝手に直さない）
4. 完了条件を自分で証明できない時は fail-close（やらずに止まって報告）

## 5. 実行前に止める条件（1つでも当てはまれば止まって祐馬さんへ）

- 💰 **お金**: freee書込・取引登録・請求・支払い
- 🗑 **削除**: ファイル・メール・ジョブの削除（アーカイブ移動はブリーフに明記があればOK）
- 📤 **外部送信**: Discord投稿・メール送信・API書込・SNS投稿
- ⚙️ **自動化の起動/停止/変更**: launchd・常駐・cron・自動実行ON
- 🔑 **認証・トークン・権限**まわり全部
- 📜 **正本ルールの書き換え**: rules.md / NOW.md / CLAUDE.md / 本ルール
- ❓ ブリーフに無い作業／完了条件を検証できない作業
- 🤔 本当に迷った時（推測で突っ走らない）

## 6. 作業後に記録する場所

| 何をしたか | どこへ |
|---|---|
| すべての作業（毎回必須） | `06_エージェント運用/00_司令塔/作業ログ_ツバキとあおい.md` |
| 自動化・ジョブを触った | `01_プロジェクト/AI自動化/導入済み.md`（**台帳更新までが1セット**） |
| 宿題・期日ができた | `task_cli.py add`（口約束禁止） |
| 祐馬さんの判断待ち | `06_エージェント運用/10_Inbox/要確認.md` |
| ルール・手順・学びが変わった | rules.md / workflows.md / learning-log.md（採用は祐馬さん承認後） |
| 現在地が変わった | `06_エージェント運用/00_司令塔/NOW.md` |

## 7. 祐馬さんに確認が必要な境界（GOが要るもの）

1. §5のストップ条件の解除すべて
2. 新しい自動化のON・既存自動化の停止・置換
3. 正本ルール文書の書き換え（＝ルールの採用・変更）
4. お金の数字に触ること全部
5. 費用が発生すること（サービス登録・購入）
6. 一発で戻せないこと（削除・外部公開・アカウント設定変更）

逆に、**GO不要で進めてよいもの**（2026-07-03 採用時の祐馬さん修正: **範囲は「祐馬さんがCodexへブリーフを貼った後・そのブリーフの範囲内」に限定**）:
- 読み取り調査・照合・レポート
- 作業ログ・台帳・タスクの記録更新
- git管理下の文書修正で、ブリーフに完了条件が明記されているもの

ブリーフを貼る**前**の段階では、Fableは「祐馬さんから直接頼まれた読み取り調査・ブリーフ／レポート作成・記録（作業ログ／task_cli／自分のメモリ）」だけを行い、頼まれていない作業はしない。

## 8. 明日からの回し方（1日の型）

```text
1. 祐馬さん → Fable 「次これ」
2. Fable: 調査 → ブリーフ作成（ここまで読み取りのみ）
3. 祐馬さん: ブリーフをCodexへ貼る（＝承認）
4. Codex: 実行 → 作業ログ記録
5. Fable: 結果を独立検証 → OKなら次の1件へ / NGなら差し戻しブリーフ
```

- 原則**1件ずつ**。並行しない（慣れるまで）
- 詰まったら「止まって報告」が正解。動いて誤魔化さない

## 9. 採用手順（このルールを正本にするには）

1. 祐馬さんが本案を読んでOKを出す
2. 最初のCodexブリーフとして「NOW.md・rules.mdへの参照1行追記＋移行設計書§8への注記」を実行（＝採用の実装もこの型で行う）
3. 以後この型で運用開始。**2週間使ってみて、合わない所を1回だけまとめて改訂**（都度いじらない）
