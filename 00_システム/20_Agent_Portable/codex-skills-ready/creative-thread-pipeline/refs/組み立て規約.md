# 組み立て規約（creative-thread-pipeline / scripts/assemble_csv.py の正本）

設計書 `00_システム/20_Agent_Portable/specs/creative-thread-pipeline-設計書.md` の §6/§7/§8 を実装ベースで集約したもの。
**スレ本文は read-only。返信先の自動導出はしない（LLM判断＝§7.1）。** ここの規約に矛盾が出たら設計書が上位。

## マニフェスト（assemble_csv.py の入力契約・正本スキーマ）
エンジンが実際に食う形＝**sections リスト**（設計§8のネスト案はこのリスト形に確定）。
`section.path` は **manifest ファイルからの相対パス**で解決する（CWD非依存）。`out_base` は絶対パス推奨。

```json
{
  "out_base": "/abs/path/2026-06-17_平将門の祟り",
  "sections": [
    { "type": "narr",   "label": "冒頭",   "path": "parts/p_bouto.md" },
    { "type": "thread", "label": "スレ①",
      "path": "thread1.md",
      "reply_map": { "2": 1, "3": 1, "12": 11, "65": 64 },
      "inserts": [
        { "after_seq": 15, "label": "解説：大蔵省", "path": "parts/kaisetsu_okurasho.md" },
        { "after_seq": 28, "label": "解説：将門の首塚", "path": "parts/kaisetsu_kubizuka.md" }
      ] },
    { "type": "narr",   "label": "前置き", "path": "parts/p_maeoki.md" },
    { "type": "thread", "label": "スレ②", "path": "thread2.md",
      "reply_map": { "...": "..." }, "inserts": [ "..." ] },
    { "type": "narr",   "label": "締め",   "path": "parts/p_shime.md" }
  ]
}
```
- `reply_map` … **seqキー:seq値**（スレ内の連番空間。rawの原スレ番号は載せない）。`>>1`バーストも明示する。
  = LLM が §6.5/6.6 で判断した返信グラフ。**script はこれをラベル化するだけ**（自動導出ではない）。
- `inserts.after_seq` … **再採番後の連番**。実在しなければ即エラー（無言スキップ禁止）。
- narr 区間（冒頭/前置き/締め/解説）は空行区切りで複数セルに分割。話者はすべて「解説者」。
- 見本: `refs/サンプル/manifest_見本.json`（平将門・実データ）＋ `refs/サンプル/平将門_台本_見本.csv`（出力見本）。

## §6.1 列構成
`登場人物, セリフ, 備考, 番号`（番号＝「連番」または「連番>>返信先連番」）。

## §6.2 構成順（備考列のセクション）
冒頭 → スレ① →〔解説×2〕→ 前置き → スレ② →〔解説×2〕→ 締め。
解説は **対象用語クラスタが一段落した直後（次の話題へ移る前）** にスレの合間へ挿入。
**「用語初出の直後」ではない**（初出は壁の途中になりがち）。位置は STEP3 で提案→確認。

## §6.3 話者
- スレタイ（各スレ1レス目）＝**男性A**。
- 以後 **男性A→女性B→男性C→女性D→男性E** をレス順にローテ（純index `ROT[i%5]`＝決定的・manifest非搭載）。
- 解説者パートは全部「解説者」。

## §6.4 大（長文ラベル）
- `>>`/`↑` 除去後の本文が **88字以上 → 大**。字数は **句読点改行を付与する前の生本文**で数える（順序＝§7.2）。
- 88は平将門のレス長分布での暫定値。**境界±5字（83〜92字）は「大候補→提案→確認」のグレーゾーン**。
- learning-log には「88」でなく「長文＝大の判定ルール」を記録。

## §6.5 下・返信先（番号列 `N>>M`）
- 返信（本文`>>`始まり / スレタイ反応 / **LLMが返信と判断したレス**）→ **下** ＋ 番号 `N>>M`（M＝返信先seq）。
- **スレタイへの反応＝`>>1`**。バースト範囲＝**概ね seq2〜5**（個別レスを引き始めた/話題が一段具体化したら終了・LLM判断→確認）。
- **下大→大（大優先）**：長文かつ返信でも「大」にし、下も>>も付けない。
- **大の直後のレスは下も>>も付けない**（返信でも素の番号）。
- ⚠️ この2つの抑制で **LLMが承認した返信の一部が出力CSVから消える**（平将門で12件＝約16%）。
  消えたリンクは §4.8 で **件数突合＆一覧可視化**（黙って消さない）。

## §6.6 返信先は「会話の流れ」準拠（最重要・平将門①66の教訓）
- 返信先は **直近 or 同一話題のwall** へ。独立トピックが完結した後に前へ戻る後方ジャンプは不自然。
- 妥当性は **§4.6 敵対監査レンズ②で LLM が本文意味から**判定。**機械の自動補正（gap≥3で寄せる）は廃止**。

## §6.7 句読点改行（TTS用）
セリフは `。、！？` の直後で改行（既存改行は一旦解除してから付与）。

## §6.8 文字コード・出荷物
- CSV＝**UTF-8(BOM付き/utf-8-sig)・LF**。
- **出荷は CSV のみ**（xlsx は廃止＝オーナー決定2026-06-20）。目視レビューは show_widget の会話ビューアを単発で出す。
- assemble_csv.py の副産物：`_台本.md`（人間可読・構成見出し付き）＋ `~/Desktop` へ CSV 複製（YMM4ドラッグ用）。

## §6.9 句点(。)・読点(、)は【あえてゲートしない】
本物2chは句点が極端に少ない＝AI臭の穴だが、**うちのCSVはTTS用に句読点改行を使う**ため句点を削ると改行が壊れる。
よって句点系の整然さは意図的にゲート対象外（将来この穴を機械で塞ごうとしない＝TTSと衝突）。

## §7.2 パイプライン順序（assemble_csv.py 内・固定）
①`>>`/`↑`除去 → ②大/下判定（**字数は改行付与前の生本文**）→ ③返信マップ反映（`N>>M`）→ ④句読点改行 → ⑤CSV/MD出力。
ラベル付与の優先順位：**`is_dai(88字+) ＞ prev_dai(大の直後) ＞ reply_map(下/>>)`**。
スレ内 raw番号（欠番あり）→seq連番の変換は **assemble_csv.py 一箇所**が read-only 入力時に機械生成。
**reply_map も after_seq も seq 空間**で扱う。平将門の「post-30一括振り直しで本文>>が壊れた」のは**スレ側を改変したから**＝二度とやらない（欠番不規則化は creative-thread-gen 側の責務）。

## §4.8 最終受入チェック（assemble_csv.py が自動出力・NGなら exit≠0）
- 解説が **指定本数ぶん発火**したか（既定＝スレ①2/スレ②2＝計4）。after_seq 実在＆正しい位置。
- **大に `>>` が付いていないか** ／ 大の直後に下・>>が無いか。
- **返信マップ突合**：`承認 reply_map − 大/直後で抑制 = 出力CSVの>>件数`。**抑制で消えたリンク一覧を可視化**。
- 返信先 `>>M` が全て **自分より前(seq)** を指すか（未来/自己参照ゼロ）。
- スレタイが各スレ先頭・男性Aか。
- 出荷物にASCII`"`が残っていないか（qaのハードゲートと整合・組み立て段でも検査）。
- 不一致が1つでもあれば **完成と言わず**該当を修正→再検証（script が exit 2 を返す）。
- ⚠️ **§4.8は「構造整合」の検査であって「会話の意味的な正しさ」は検査しない**（返信マップ突合は `承認−抑制=出力>>` の恒等式＝reply_mapが間違っていても突合はPASSする）。**返信先が会話として噛み合うか／後方ジャンプの妥当性は STEP4.5 の敵対監査（LLMレンズ②）が担保する**。§4.8 PASS＝出荷可ではなく、STEP4.5通過とセットで初めて出荷可。
