# YouTube台本制作パイプライン

2ch歴史チャンネルの動画制作を6ステップで効率的に進めるワークフロー。

## 🧠 学習ループ（2026-06-07導入・必須）

**執筆前（毎回）:**
1. `/Users/kojinn/2nd-Brain/03_知識ベース/YouTube・コンテンツ制作/台本執筆ルール.md` を読み、R1〜のルールと修正パターンを台本に反映する
   - 特にR1: 冒頭は30秒以内に本題（挨拶・前置きを削る）
   - R2: 50%地点に中盤フックを配置、スレは数より密度

**制作完了後（毎回）:**
2. ユーザーから受けた修正指示・初稿からの変更点を
   `2nd-Brain/03_知識ベース/YouTube・コンテンツ制作/制作ログ/YYYY-MM-DD_ネタ名.md` に保存する
   （何を直されたか・なぜ・次回どうするかの3点）
3. 同じ指摘が2回以上出ていたら、台本執筆ルール.md の「修正パターン」セクションに追記する

**月次（自動）:** launchd `com.claude.script-learning` が離脱データ再分析＋制作ログのルール昇格を実行する。

## 🧬 スレコーパス蓄積 ＆ オリジナルスレ生成（2026-06-07導入）

**スレをスクレイピングするたび（必須）:**
- thread_scraper.py の出力を必ず
  `2nd-Brain/03_知識ベース/YouTube・コンテンツ制作/スレコーパス/YYYY-MM-DD_テーマ名.md` にも保存する
  （冒頭に `<!-- 採用: スレ① / 未採用 -->` 等のメタコメントを付ける）
- コーパスが10本溜まるごとに `2chスレ文体パターン.md` の「学習メモ」を更新する
  （レスのリズム・新しい言い回し・伸びる流れのパターンを抽出して追記）

**オリジナルスレ生成モード（実スレが不足するネタ用）:**
1. 発動条件: 実スレが3本未満、またはユーザーが「オリジナルで」と指定したとき
2. `2chスレ文体パターン.md` を読み、定番構造（イッチ→懐疑→知識ニキ→脱線→収束）と口調比率に従って生成する
3. `スレコーパス/` から近いテーマのスレを1〜2本読み、リズムの参考にする（コピーはしない）
4. **歴史事実は必ずWebSearchでファクトチェック**（youtube-script-checkerの第3段階と同基準）。諸説あるものは「〜説もあるで」とスレ内で明示
5. 台本の備考列には「スレ①(オリジナル)」と明記し、スプシのO列（スレ可用性）に「オリジナル生成」と記録する

**メリット**: ネタ選択の自由度が上がる（スレ可用性✕でも作れる）＋転載権利リスクゼロ。
コーパスが溜まるほど「本物らしさ」が上がる学習構造。

## 使い方

ユーザーがテーマを指定したら、以下の6ステップを順番に実行する。
各ステップ完了後にユーザーに確認を取ってから次へ進む。

## 台本フォーマット仕様（全ステップ共通）

台本は以下の **4列スプレッドシート形式** で作成する。
全パーツ（冒頭・スレコメント・解説・前置き・締め）をこのフォーマットに統一する。

### 列構成

| 列名 | 内容 | 例 |
|---|---|---|
| 登場人物 | 発言者の識別名 | 解説者、男性A大、女性B、男性C下 |
| セリフ | 発言内容（セル内改行あり） | 幕末の英傑で打線組んだで！ |
| 備考 | セクション名 | 冒頭、スレ①、解説：小松帯刀、前置き、スレ②、締め |
| 番号 | セクション内の通し番号 | 1, 2, 3... |

### キャラ割り当てルール

**基本ローテーション（5人制）：**
男性A → 女性B → 男性C → 女性D → 男性E → 男性A → 女性B → ...（ループ）

**特殊サフィックス（2026-06-21 creative-thread-pipelineと規約統一・エンジン共有）：**

| サフィックス | 意味 | 判定 |
|---|---|---|
| 大 | **88字以上の長文壁**（知識ニキの長文解説など・1スレに複数可） | 本文（>>/↑除去後）が88字以上 |
| 下 | 返信。番号列に `N>>M`（M=返信先のスレ内連番seq）を付ける | reply_map（LLM判断）に従う |
| （なし） | 通常発言・短レス | 上記以外 |

- **スレタイ（各スレ1レス目）＝男性A・サフィックスなし**（旧「大＝イッチ」から変更。イッチに大は付けない）。
- **優先順位：大(88字+) ＞ 大の直後 ＞ 下/>>**。下大→大（大優先＝大には下も>>も付けない）。大の直後のレスも下/>>なし。
- これらは**手作業でなく `assemble_csv.py` が機械適用**する（STEP2-4）。**正本＝`~/.claude/skills/creative-thread-pipeline/refs/組み立て規約.md`**（実スレ版・創作版の共通正本）。

**解説者：**
- 冒頭パート、解説パート、前置きパート、締めパートは全て「解説者」
- 解説者の複数行は連続して配置する（解説者が3行連続など）

### セクション構成（備考列）

台本のセクションは以下の順で構成する：

| 順序 | 備考列の値 | 内容 |
|---|---|---|
| 1 | 冒頭 | 動画冒頭の掴み（解説者） |
| 2 | スレ① | 前半のスレコメント群 |
| 3 | 解説：〇〇 | 解説パート（必要な分だけ挿入。スレの合間に配置可能） |
| 4 | 前置き | 前半→後半の転換（解説者） |
| 5 | スレ② | 後半のスレコメント群 |
| 6 | 解説：〇〇 | 後半の解説パート（必要に応じて） |
| 7 | 締め | 動画の締めくくり（解説者） |

- 解説パートの挿入位置は、**該当用語クラスタが一段落した直後（次の話題へ移る前）**に配置する（用語初出の直後ではない＝壁の途中を避ける）
- 解説が3本ある場合、スレ①内に2本、スレ②内に1本のように分散配置してよい
- 備考列の解説名は「解説：人物名」「解説：事件名」のように具体名を入れる

## STEP 2: 台本パーツ生成
**使用スキル**: `/youtube-script-parts`（2-3で使用。2-1はthread_scraper.py、2-2/2-4はパイプライン独自処理）

STEP1で決まった構成に基づき、以下を実行:

### 2-1: スレコメント自動取得（thread_scraper.py v2）
**`thread_scraper.py` を使って、まとめサイトからスレタイ＋元スレのレスのみを自動取得する。**

**スクリプトの探し方（必ずこの順序で実行）：**
1. `Glob` ツールで `**/thread_scraper.py` を検索する
2. 見つかったら、そのパスを使って実行する
3. 見つからない場合は、`Read` ツールでこのスキルのベースディレクトリ内の `thread_scraper.py` を読み取り、作業ディレクトリにコピーしてから実行する
4. それでも見つからない場合のみ、手動コピペにフォールバックする

**取得対象：**
- スレタイ（記事タイトル）
- 元2chスレのレス（まとめサイトの記事本文部分）

**取得しないもの：**
- まとめサイトのコメント欄（読者コメント）は一切取得しない
- 広告・ナビゲーション等のノイズテキスト

**重要：WebFetchは使わない。** WebFetchはJSレンダリング不可で全件取得できないリスクがある。
代わりに `thread_scraper.py`（requests + BeautifulSoup）を使う。こちらはまとめサイトの
HTML構造をサイト別パーサーで解析し、元スレのレスだけを構造化抽出する。

**対応サイト：**
- livedoor Blog系（absurd.blogo.jp、gabareki.blog.jp、tihourekisimatome.blog.jp 等）
- nanjgod.com
- FC2系
- その他主要2chまとめサイト

**手順：**
1. STEP1で選定したスレのURLを確認する
2. スクリプトを探す（上記「スクリプトの探し方」に従う）
3. `thread_scraper.py` を実行してレスを一括取得する：
   ```bash
   python /path/to/found/thread_scraper.py \
     "URL1" "URL2" "URL3" "URL4" \
     -o scraped_threads.md
   ```
4. **取得もれチェックを確認する**（スクレイパーが自動で実行）：
   - 出力冒頭の「取得サマリー」でスレごとの取得件数・文字数・検証結果を確認
   - 各スレ内の「取得もれチェック」でレス番号の欠番がないか確認
   - ✅ OK → そのまま進む
   - ⚠ WARN（欠番あり/件数少） → ユーザーに報告し、手動確認を依頼
   - ⛔ FAIL（取得ゼロ） → フォールバック（手動コピペ）に切り替え
5. 取得したレスを **「番号: 本文」形式のスレファイル**として保存する（欠番・>>はそのまま保持）。
   - ⚠️**キャラ割り当て・大/下・>>の付与はSTEP2-4で `assemble_csv.py` が機械適用するので、ここでは手で付けない**（旧：手で大/下を振っていた工程は廃止）。
   - ASCII `"` は `「」` に変換しておく（共有エンジンのハードゲート対象）。
   - >>参照はスレ本文のまま残してよい（reply_mapのシード元になる）。
   - 削除提案（カテゴリA:完全脱線、B:テーマ隣接だが冗長）を**文字数付き**で提示 → 【確認ポイント①】（削除はスレ本文の行単位。欠番が増えてもエンジンが連番(seq)へ再採番するので問題ない）

**⚠ レスの削除・追加に関する絶対ルール：**
- **自動削除禁止**: レスを勝手に削除してはならない。必ず削除提案をユーザーに提示し、承認を得てから削除する。
- **自動追加禁止**: オリジナルコメントや導線コメントを勝手に追加してはならない。必ず追加提案をユーザーに提示し、承認を得てから追加する。
- 提案時は「どのレスを」「なぜ」削除/追加するかを明示し、文字数への影響も記載する。
- ユーザーが「OK」「承認」等と明示的に回答するまで、一切の削除・追加を実行しない。

**取得失敗時のフォールバック：**
サイト構造変更等でスクレイパーが取得に失敗、または取得もれチェックでFAILの場合：
```
スクレイパーで取得できなかったスレがあります。以下のURLを開いて、
記事本文（元スレのレス部分のみ）をコピペしてください。
※まとめサイト下部のコメント欄は不要です。
① [URL]
```

**⚠ 絶対禁止：取得失敗時にスレを差し替えない**
取得に失敗したスレがあっても、STEP1で決定した構成（前半/後半のスレ選定）は絶対に変更しない。
別のスレを探しに行ったり、構成を組み直したりしてはならない。
取得できなかったスレはユーザーに手動コピペを依頼し、同じスレで台本制作を続行する。
理由：構成はSTEP1で吟味して決定済みであり、取得技術の問題で内容を変えるべきではない。

### 2-2: 解説対象の選定 → ユーザーに提案
台本に登場する用語・人物・事件から解説パートを入れるべきものを選定し、ユーザーに提案する。

**解説対象の選定基準：**
- **解説すべき**: 教科書に載っていないレベルの人物・合戦・制度（例：尼子氏、耳川の戦い、雑賀衆）
- **解説不要**: 中学の教科書に出てくるレベルの人物（例：織田信長、武田信玄、豊臣秀吉）
- **判断基準**: 「この用語を知らなくても前後の文脈で理解できるか？」→ 理解できない場合は解説を入れる

提示フォーマット:
```
解説を入れるべき候補:
| # | 用語 | 登場箇所 | 理由 |
入れなくてよいもの:
| 用語 | 理由 |
```

### 2-3: 台本パーツ生成
**使用スキル**: `/youtube-script-parts` のリファレンスに従って生成

- 冒頭パート（掴み）→ 登場人物：解説者、備考：冒頭
- 解説パート（選定基準に基づき2〜3本）→ 登場人物：解説者、備考：解説：〇〇
- 前置きパート（前半→後半の転換）→ 登場人物：解説者、備考：前置き
- 締めパート（まとめ＋チャンネル登録誘導）→ 登場人物：解説者、備考：締め

### 2-4: 台本組み立て → CSV（共有エンジン assemble_csv.py・2026-06-21一本化）
全パーツ（冒頭＋スレ①＋解説＋前置き＋スレ②＋解説＋締め）を **manifest(JSON) に並べ、`~/.claude/skills/creative-thread-pipeline/scripts/assemble_csv.py` でCSVに機械組み立て**する（手結合の`「セリフ」`md・STEP4のopenpyxl変換は廃止＝実スレ版・創作版で同一エンジン）。
1. **返信マップ(reply_map)**：`creative-thread-pipeline/scripts/seed_reply_map.py <スレ.md>` で本文の明示`>>`をraw→seq自動シード（実スレは明示>>が多いので自動捕捉率が高い）→暗黙の返信だけLLMが組み立て規約 §6.5/6.6 で足す。
2. **解説の挿入位置(after_seq＝クラスタ末尾)・採用版**を提案→確認。
3. manifest（スキーマ＝`creative-thread-pipeline/refs/組み立て規約.md`）を書き、`python3 assemble_csv.py manifest.json` を実行。
   - 出力＝**CSV(utf-8-sig/句読点改行)＋_台本.md＋~/Desktop複製**。
   - **§4.8受入チェック＋安価率＋大グレーゾーン**が自動表示（NGならexit2＝完成と言わせない）。話者ローテ・大/下・>>・解説挿入は全部エンジンが付ける。

### 文字数ルール（重要）
- **スレコメント部分：6,000文字以上**
- **＋解説者パーツ（冒頭・解説・前置き・締め）を合わせて合計7,000文字超**
- 超過する場合はカテゴリBから削除候補を**提案**し、ユーザー承認後に削除する（自動削除禁止）
- 不足する場合はオリジナル導線コメントの追加を**提案**し、ユーザー承認後に追加する（自動追加禁止）

**完了条件**: ユーザーが台本全体を承認

## STEP 3: 3段階チェック（ポリシー＋文脈＋ファクトチェック）
**使用スキル**: `/youtube-script-checker`

STEP2-4で組み立て済みの台本（`<base>_台本.md`／CSV）に対して **3段階** のチェックを実行する（指摘が出たらスレ本文/partsを直して `assemble_csv.py` を再実行）。
全段階を通過するまでSTEP 4には進まない。

### 第1段階：自動8層スキャン（既存）
`scripts-v2/scan_v4.py` による自動チェック：
- L1: NGワード直接マッチ（ng-dictionary.md準拠）
- L2: 差別的トロープパターン（ng-dictionary.md準拠）
- L3: 悪意ある表現パターン（ng-dictionary.md準拠）
- L4: 誤検知除外（ng-dictionary.md準拠）
- L5: 辞書外マニュアル検知
- L6: 頻度カウンター
- L7: メタデータチェック
- L8: 保護対象グループ文脈分析
- 伏字チェック（〇○●■□等）

🔴ヒットは言い換え候補を自動提案 → ユーザー承認 → 一括置換 → 再スキャン（🔴ゼロまで）

### 第2段階：LLM文脈チェック
自動スキャンでは検出できない「文脈依存」の問題を、LLMが台本全文を通して判定する。

**チェック項目：**

| # | カテゴリ | チェック内容 | 例 |
|---|---------|------------|---|
| 1 | 死の肯定・美化 | 死を賞賛・憧れの対象として描く文脈 | 「〇〇の最期は美しい」→ 言い換え |
| 2 | グラフィック暴力 | 処刑・戦闘の過度に詳細な描写 | 臓器・出血量の具体描写 → 簡潔に |
| 3 | 人物への繰り返し侮辱 | 同一人物を繰り返し嘲笑・蔑む | 3回以上の身体的特徴揶揄 → 削減 |
| 4 | 政治的・宗教的煽動 | 現代の政治問題に直結する表現 | 特定国家・民族の優劣 → 客観記述に |
| 5 | 収益化リスク表現 | YouTube AI が広告非適合と判定しうる表現 | 暴力＋笑い、いじめの正当化 → 修正 |
| 6 | ユーザー指摘フィードバック | 過去にユーザーが「これ大丈夫？」と指摘した表現パターン | refs/channel-rules.md の蓄積を参照 |

**判定基準：**
- 🔴（要修正）: 収益化停止・警告リスクが高い
- 🟡（要注意）: グレーゾーン、ユーザーに判断を委ねる
- ✅（問題なし）: 歴史的文脈として許容

### 第3段階：歴史的事実ファクトチェック
解説パート・冒頭パートを中心に、**事実の正確性**をWebSearchで検証する。
視聴者に誤った情報を与えないための最終チェック。

**チェック対象：**
- 冒頭パート（解説者のセリフ全体）
- 解説パート（解説：〇〇 のセリフ全体）
- 前置きパート（解説者のセリフ全体）
- 締めパート（解説者のセリフ全体）
- スレコメント内で解説者が追記した導線コメント

**チェック項目：**

| # | カテゴリ | 内容 | 例 |
|---|---------|------|---|
| 1 | 漢字の誤り | 人名・地名・用語の漢字が正しいか | 伊達政宗→✅、伊達政宗→✅、伊達正宗→❌ |
| 2 | 年代・年号 | 事件・戦いの年代が史実と一致するか | 関ヶ原の戦い＝1600年→✅、1603年→❌ |
| 3 | 人物の年齢・生没年 | 記載された年齢が生没年と矛盾しないか | 「享年49歳」→ 生没年から逆算して検証 |
| 4 | 史実との矛盾 | 事実関係が通説と異なっていないか | 「信長が関ヶ原で戦った」→❌（本能寺の変で既に死去） |
| 5 | 因果関係の誤り | 原因と結果の関係が正しいか | 「〇〇の戦いに勝ったことで△△になった」→ 検証 |
| 6 | 人物関係の誤り | 親子・主従・同盟関係が正しいか | 「〇〇の息子である△△」→ 検証 |
| 7 | 官職・役職名 | 人物の官職・役職が正しいか | 「〇〇守護」→ 正しい官職か検証 |

**手順：**
1. 台本から解説者パート（冒頭・解説・前置き・締め）を抽出する
2. 各パートに含まれる **人名・地名・年代・事件名・人物関係** を全てリストアップする
3. WebSearchで各項目を検証する（Wikipedia・歴史辞典等を参照）

**概要欄ルール（2026-06-21 実データ117本分析で確定・平将門がテンプレ基準）：**
- 📏**短く＝可変部 約150〜250字**（旧500字は長すぎ。実チャンネルの大ヒット群は169〜224字）。見どころの羅列・スレ名言の貼り付けは削ってフックだけ残す。
- 🎙️**"解説者の声"（である体）で統一**：「〜だろう」「果たして〜のだろうか」「議論を見ていこう」。動画冒頭ナレと地続きにする。
- 🚫**AI広告コピー語を使わない**：その正体に迫る／〜を掘り下げます／真剣に議論しました／〜間違いなしです／em-dash(——)／体言止めの羅列。これらは全部AI臭。
- 1行目はタイトル冒頭キーワードと一致（SEO）。「2ちゃんねるの歴史オタクたちの議論を見ていこう」の文脈は必ず入れる。
- 固定ブロック（チャンネル説明・素材クレジット）は一字一句変えない。
- **2つの実フォーマット**（任意で選択）：A型＝上記の短い解説者シノプシス（直近の標準）／B型＝先頭にハッシュタグ5個＋「↓↓人気の動画↓↓」＋過去の伸びた動画リンク2〜3本（古い大ヒット群の型・回遊狙い）。A＋B併用も可。
- 📊**タイトルは neta-research の `predict_score.py` で実データ採点してから決める**（過去109本DB）。単独人物は中央値≈6k＝「実は/たった1人で」等の意外性フックで上振れを狙う。サムネ文言はタイトルと重複させず別フックを絵で見せる。

### ハッシュタグ
**15個**生成する（YouTubeの推奨上限）。

選定方針：
1. **テーマ直結**（人名・事件名）→ 2〜3個
2. **2ch/ゆっくり解説系**（#2ch歴史 #ゆっくり解説）→ 必須2個
3. **時代・カテゴリ**（#戦国時代 #日本史 #歴史解説）→ 2〜3個
4. **関連人物**（台本に登場する主要人物）→ 2〜3個
5. **大河ドラマ連動**（放映中・直近の大河で関連テーマがある場合）→ 1〜2個
   - 連動できる大河がない場合はスキップし他のタグで補う
6. **その他関連キーワード**（合戦名・政策名など）→ 残り枠
