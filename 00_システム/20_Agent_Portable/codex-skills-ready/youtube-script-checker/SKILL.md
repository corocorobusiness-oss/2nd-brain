# YouTube台本3段階チェッカー

2ch歴史チャンネル「2chで世界史学ぶ民」専用の台本監査スキル。
**3段階チェック**で、ポリシー違反・文脈リスク・事実誤認をゼロに近づける。

- **第1段階**: 自動8層スキャン（NGワード・差別・暴力・伏字等）
- **第2段階**: LLM文脈チェック（自動スキャンで捕れない文脈依存の問題）
- **第3段階**: 歴史的事実ファクトチェック（漢字・年代・史実の正確性）

---

## 手順

### Step 0: 準備

1. `refs/channel-rules.md` を読み込む（チャンネル固有の絶対禁止ルール）
2. `refs/ng-dictionary.md` を読み込む（8層辞書の全データ）
3. `refs/youtube-6policy.md` を読み込む（YouTube公式ポリシー判定基準）
4. `refs/replacement-db.md` を読み込む（言い換えデータベース）

### Step 1: 台本取得

ユーザーから台本を受け取る。以下のいずれかの形式に対応：

**A. Google Sheets URL**
```
https://docs.google.com/spreadsheets/d/{id}/edit?gid={gid}
```
→ gviz APIでCSV取得:
```
https://docs.google.com/spreadsheets/d/{id}/gviz/tq?tqx=out:csv&gid={gid}
```

**B. テキスト直接貼り付け**
→ そのままスキャン対象とする

**C. タイトル・説明文・タグも取得**（ユーザーが提供した場合）
→ メタデータとして別途スキャン

## ═══════════════════════════════════════
## 第1段階：自動8層スキャン
## ═══════════════════════════════════════

### Step 2: 8層スキャン実行

`scripts-v2/scan_v4.py` を実行する。
**CRITICAL**: scan_v4.py は `refs/data/*.tsv` から全データを動的にロードする。Pythonコード内にNGワードのハードコードは一切ない。

以下の8層を順番にスキャンする：

**L1: NGワード直接マッチ（`refs/data/l1_words.tsv` から動的ロード）**
- 漢字・ひらがな・カタカナの全表記揺れを検知（例：死ぬ/しぬ/シヌ、バカ/ばか）
- ただし別の用語の一部として含まれる場合は誤検知除外で対応（例：シヌクレイン）
- カテゴリ: ヘイト、障害者差別、性差別、LGBTQ差別、侮辱、暴力、性的表現、下品表現、グラフィック暴力、児童安全、CH固有禁止

**L2: 差別的トロープパターン（`refs/data/l2_patterns.tsv` から動的ロード）**
- 「〇〇人は△△」型のステレオタイプ構文
- 保護対象グループ＋動物/害虫/病気比喩（非人間化）
- 民族/人種/性別＋優劣判定パターン

**L3: 悪意ある表現パターン（`refs/data/l3_patterns.tsv` から動的ロード）**
- 暴力行為＋肯定/称賛/美化パターン
- 排除・隔離の正当化
- 陰謀論的主張

**L4: 誤検知除外（`refs/data/l4_false_positives.tsv` から動的ロード）**
- 動詞活用による誤検知（考えた≠穢多 等）
- 地名・人名・歴史用語
- 2chニックネーム（フロカス＝フロイス 等）
- 死の複合語（必死、決死 等）
- 殺の複合語（必殺、殺到 等）
- ひらがな・カタカナ表記の一般語（シヌクレイン 等）

**L5: 辞書外マニュアル検知（`refs/data/l5_manual.tsv` から動的ロード）**
- Layer A: 侮辱・蔑称・下品
- Layer B: 性的・ジェンダー
- Layer C: 民族・国際的敏感
- Layer D: YouTube AI特別検知

**L6: 頻度カウンター**
- 同一の🟡表現が5回以上出現 → 「繰り返し使用」として🔴に昇格
- 下品表現の総出現回数をカウント

**L7: メタデータチェック**（タイトル・説明文・タグが提供された場合）
- タイトルに冒とく的表現がないか
- 説明文にNGワードがないか
- センセーショナルなタイトル検知

**L8: 保護対象グループ文脈分析**
- YouTube保護対象: 年齢、民族/人種、障がい、国籍、宗教、性別/ジェンダー/性的指向
- これらに対する「暴力助長」「憎悪扇動」「非人間化」「優劣主張」を構文レベルで検知
- 同一人物への身体的侮辱の繰り返し検知

### Step 3: 伏字チェック

〇○◯●■□★☆＊＿ を使った全ての伏字パターンを検知。
YouTube AIは伏字でも元の単語を推定するため、全て言い換え必須。

検知パターン:
- `[文字][〇○◯●■□][文字]` の組み合わせ
- 既知の伏字パターン（ガ〇ジ、ホ〇、ク〇、ア〇、バ〇、デ〇、カッ〇 等）
- 半角カナ伏字（ﾊﾅｸ〇ﾎｼﾞー 等）

### Step 4: 6ポリシー判定

各ヒットをYouTube6ポリシーに照合して3段階判定：

| レベル | 意味 | 対応 |
|---|---|---|
| 🔴 要修正 | ストライク/警告リスク有り | 必ず言い換え |
| 🟡 要注意 | グレーゾーン・文脈次第 | 言い換え推奨 |
| ✅ 問題なし | 歴史文脈/チャンネル文体として許容 | そのまま |

判定基準は `refs/youtube-6policy.md` に従う。

### Step 5: 言い換え自動提案

🔴と🟡のヒットに対して `refs/replacement-db.md` から言い換え候補を自動マッチング。

言い換えルール:
- 2ch風チャンネルの口調を維持する
- 歴史的正確性を損なわない
- 不自然にならない程度に安全側に寄せる
- refs/channel-rules.md の絶対禁止ワード全件は言い換え後のテキストにも使わない

### Step 6: ユーザー確認（第1段階）

修正一覧表を提示してユーザーの承認を待つ。

表示フォーマット:
```
【第1段階：自動8層スキャン結果】
| # | カテゴリ | 修正前 | 修正後 | 理由 | リスク |
|---|---------|--------|--------|------|--------|
| 1 | ヘイト  | ガ〇ジ | 変人   | 障害者差別語 | 🔴 |
```

### Step 7: 一括置換 → 再スキャン

1. ユーザー承認後、全修正を一括適用
2. Step 2〜4を再実行（再スキャン）
3. 🔴ゼロになるまでループ（最大3回）
4. 最終確認で以下を検証:
   - 伏字ゼロ
   - 禁止ワードゼロ（refs/channel-rules.md の絶対禁止ワード全件）
   - L1〜L8の🔴ゼロ

**第1段階の完了条件**: 🔴ゼロ・伏字ゼロ・禁止ワードゼロ

## ═══════════════════════════════════════
## 第2段階：LLM文脈チェック
## ═══════════════════════════════════════

第1段階の自動スキャンでは検出できない「文脈依存」の問題を、LLMが台本全文を通して判定する。

### Step 8: 文脈チェック実行

台本全文を読み通し、以下の7カテゴリについて問題がないか判定する。

**チェック項目：**

| # | カテゴリ | チェック内容 | 判定基準 |
|---|---------|------------|---------|
| 1 | 死の肯定・美化 | 死を賞賛・憧れの対象として描く文脈がないか | 「最期は美しい」「見事な最期」→ 🔴。客観的な「〇〇で命を落とした」→ ✅ |
| 2 | グラフィック暴力 | 処刑・戦闘の過度に詳細な描写がないか | 臓器・出血量の具体描写 → 🔴。「戦いで敗北した」→ ✅ |
| 3 | 人物への繰り返し侮辱 | 同一人物を繰り返し嘲笑・蔑んでいないか | 同一人物への身体的特徴揶揄が3回以上 → 🔴 |
| 4 | 政治的・宗教的煽動 | 現代の政治問題に直結する表現がないか | 特定国家・民族の優劣判定 → 🔴。歴史的事実の客観記述 → ✅ |
| 5 | 収益化リスク表現 | YouTube AIが広告非適合と判定しうる表現がないか | 暴力＋笑い、いじめの正当化、自傷行為の描写 → 🔴 |
| 6 | ユーザー指摘フィードバック | `refs/channel-rules.md` に蓄積された過去のユーザー指摘パターンに該当しないか | 過去に「これ大丈夫？」と指摘された表現パターン → 🟡 |
| 7 | 現代の実在個人・団体への言及 | 戦後〜存命の実在個人（芸能人・有名人・政治家・経営者等）や現存企業を**実名**で挙げていないか。特に負の逸話（祟り・不幸・事故・干された・トラブル・スキャンダル）と結びつけていないか | 現代の存命個人を実名＋負の逸話に結びつける → 🔴（名誉毀損・規約抵触リスク。実名を外し匿名化「売れる前の若手芸人が〜」）。現存企業＋負の都市伝説 → 🟡。物故した歴史上の公人（例：戦前の大臣）の客観記述 → ✅ |

> 補足：creative-thread-gen由来のスレ本文は2ch風の軽口が混ざるため、ここを特に注意して読む。2026-06-19にこのカテゴリ欠落で「太田光（実在芸人）が首塚で祟られ仕事が干された」が機械ゲート・旧LLMチェック双方をすり抜けた実績あり（→匿名化で修正）。

**判定基準：**
- 🔴（要修正）: 収益化停止・警告リスクが高い → 言い換え必須
- 🟡（要注意）: グレーゾーン → ユーザーに判断を委ねる
- ✅（問題なし）: 歴史的文脈として許容

### Step 9: 文脈チェック結果の提示

```
【第2段階：LLM文脈チェック結果】
| # | カテゴリ | 該当箇所 | 問題の内容 | 修正案 | リスク |
|---|---------|---------|-----------|--------|--------|
| 1 | 死の美化 | 締め3行目 | 「見事な最期」が死の美化に該当 | 「壮絶な生涯を全うした」 | 🔴 |
```

### Step 10: 文脈チェック修正 → 確認

1. ユーザー承認後、修正を適用
2. 修正箇所を第1段階（自動スキャン）に再通して矛盾がないことを確認

**第2段階の完了条件**: 🔴ゼロ（🟡はユーザー判断済み）

## ═══════════════════════════════════════
## 第3段階：歴史的事実ファクトチェック
## ═══════════════════════════════════════

解説パート・冒頭パートを中心に、**事実の正確性**をWebSearchで検証する。
視聴者に誤った情報を与えないための最終チェック。

### Step 11: ファクトチェック対象の抽出

台本から以下のパートを抽出する（解説者が書いた部分が対象）：

**チェック対象：**
- 冒頭パート（解説者のセリフ全体）
- 解説パート（解説：〇〇 のセリフ全体）
- 前置きパート（解説者のセリフ全体）
- 締めパート（解説者のセリフ全体）
- スレコメント内で解説者が追記した導線コメント（ある場合）

**チェック対象外：**
- スレコメント部分（2chの住民の発言はチェック対象外。ただし解説者が追記した導線コメントは対象）

### Step 12: チェック項目の洗い出し

抽出したパートから、以下のカテゴリに該当する情報を全てリストアップする：

| # | カテゴリ | 内容 | 例 |
|---|---------|------|---|
| 1 | 漢字の誤り | 人名・地名・用語の漢字が正しいか | 伊達政宗→✅、伊達正宗→❌ |
| 2 | 年代・年号 | 事件・戦いの年代が史実と一致するか | 関ヶ原の戦い＝1600年→✅、1603年→❌ |
| 3 | 人物の年齢・生没年 | 記載された年齢が生没年と矛盾しないか | 「享年49歳」→ 生没年から逆算して検証 |
| 4 | 史実との矛盾 | 事実関係が通説と異なっていないか | 「信長が関ヶ原で戦った」→❌ |
| 5 | 因果関係の誤り | 原因と結果の関係が正しいか | 「〇〇の勝利により△△が実現した」→ 検証 |
| 6 | 人物関係の誤り | 親子・主従・同盟関係が正しいか | 「〇〇の息子である△△」→ 検証 |
| 7 | 官職・役職名 | 人物の官職・役職が正しいか | 「〇〇守護」→ 正しい官職か検証 |

### Step 13: WebSearchによる検証

**手順：**
1. リストアップした各項目について WebSearch で検索する
2. Wikipedia・コトバンク・国史大辞典等、信頼性の高い情報源を参照する
3. 複数の情報源で裏取りする（1つの情報源だけで判断しない）
4. 検索結果から正確な情報を特定し、台本の記述と照合する

**検証の優先順位（リソース効率のため）：**
1. **最優先**: 人名の漢字、年代・年号（間違いが目立ちやすく視聴者から指摘されやすい）
2. **高**: 生没年・享年、人物関係（親子・主従等）
3. **中**: 因果関係、官職・役職名
4. **低**: すでに検証済みの一般常識レベルの事実

### Step 14: ファクトチェック結果の提示

```
【第3段階：歴史的事実ファクトチェック結果】
| # | 箇所 | チェック項目 | 台本の記述 | 正しい情報 | 判定 | 出典 |
|---|------|------------|-----------|-----------|------|------|
| 1 | 冒頭 | 年代 | 1598年に関ヶ原... | 1600年が正確 | ❌要修正 | Wikipedia |
| 2 | 解説：小松帯刀 | 享年 | 享年36歳 | 享年36歳（1835-1870） | ✅正確 | コトバンク |
| 3 | 解説：小松帯刀 | 漢字 | 小松帯刀 | 小松帯刀 | ✅正確 | Wikipedia |
| 4 | 冒頭 | 人物関係 | 西郷の盟友 | 大久保利通の方が適切 | 🟡要確認 | 国史大辞典 |
```

判定の3段階：
- ❌ 要修正：明確な事実誤認 → 必ず修正
- 🟡 要確認：諸説あり/微妙な表現 → ユーザーに判断を委ねる
- ✅ 正確：検証済みで正しい

### Step 15: ファクトチェック修正 → 再検証

1. ❌の項目を修正する（正しい情報で置き換え）
2. 修正後のテキストで再度ファクトチェックを実行
3. ❌ゼロになるまでループ
4. 修正箇所が第1段階・第2段階のチェックに引っかからないことを確認

**第3段階の完了条件**: ❌ゼロ（全ファクト✅または🟡ユーザー確認済み）

### Step 16: 出力

- 最終行に合格証明（「3段階チェック完了・🔴ゼロ・伏字ゼロ・❌ゼロ」）

## チャンネル固有ルール（クイックリファレンス）

詳細は `refs/channel-rules.md` を参照。ここでは最重要ルールのみ記載。

### 絶対禁止ワード
refs/channel-rules.md に記載の全禁止ワードを参照すること（常に最新のリストを使う）。

### 伏字全排除
〇○を使った伏字は全て言い換えに変換。伏字のまま残さない。

### 変換必須パターン
- 動物蔑称（猿/犬で人を呼ぶ）→ 太閤/大御所等の歴史呼称に変換
- ジェンダーステレオタイプ（「女性は〇〇」型）→ 個人名に変換
- 女性物化表現 → 中立表現に変換
- 身体侮辱（デブ、ハゲ等）→ 中立表現に変換

### 許容される表現（チャンネル文体）
こいつ/コイツ、ンゴ、草、ぐう〇〇、すこ、ワイ等の2ch口調はチャンネルの基本文体として許容。
処刑、切腹、朝鮮出兵、蝦夷地等の歴史用語は客観記述として許容。

## 例

### 例1: Google Sheetsから台本3段階チェック
ユーザーが言う：「この台本チェックして https://docs.google.com/spreadsheets/d/xxx」
アクション：
1. gviz APIでCSV取得
2. 第1段階：8層スキャン → 言い換え提案 → ユーザー承認 → 一括置換 → 再スキャン
3. 第2段階：LLM文脈チェック → 問題箇所の提示 → ユーザー承認 → 修正
4. 第3段階：歴史ファクトチェック → WebSearch検証 → 結果提示 → 修正
5. 修正済みCSV + 言い換えリストCSV + ファクトチェックCSV + 監査レポートCSVを出力
結果：3段階チェック済みのクリーンな台本CSV4点セット

### 例2: テキスト直接貼り付け
ユーザーが言う：「台本チェックして（テキスト貼り付け）」
アクション：
1. テキストを直接スキャン
2. 以降は例1と同じ
結果：3段階チェック済みのクリーンな台本CSV4点セット

### 例3: ファクトチェックのみ
ユーザーが言う：「この解説文のファクトチェックして」
アクション：
1. 第3段階のみ実行（Step 11〜15）
2. ファクトチェックレポートCSVを出力
結果：ファクトチェックレポート1点

## トラブルシューティング

### Google SheetsのCSV取得に失敗する場合
原因：シートが非公開、またはgidが間違っている
解決策：シートの共有設定を「リンクを知っている人全員が閲覧可」にしてもらう

### 誤検知が多すぎる場合
原因：L4（誤検知除外）のパターンが不足
解決策：`refs/data/l4_false_positives.tsv` に除外パターンを追加

### 言い換え候補が見つからない場合
原因：`refs/replacement-db.md` に該当エントリがない
解決策：文脈に応じて手動で言い換えを提案し、refs/replacement-db.mdに追加

### WebSearchでファクトチェックの情報が見つからない場合
原因：マイナーな人物・事件で信頼できる情報源がない
解決策：ユーザーに「この情報の出典を確認できますか？」と質問する。
検証できない情報には「※出典未確認」と注記し、🟡判定とする。

### ユーザーの過去指摘が channel-rules.md にない場合
原因：フィードバックDBへの追記が漏れている
解決策：ユーザーが「これ大丈夫？」と指摘した表現は、チェック完了後に
`refs/channel-rules.md` に追記する（次回以降の第2段階で自動検知されるようになる）。

### 設計思想

チェッカーの全データ（NGワード、パターン、誤検知除外）は **TSVファイルに外出し** されている。
scan_v4.py 内にワードのハードコードは一切ない。

これにより：
- 新しい単語やパターンを追加するとき、**TSVファイルに1行追加するだけ**で即反映
- Pythonコードの修正は不要
- 語数が増えても設計が破綻しない

### ファイル構成

```
refs/
  data/
    l1_words.tsv              ← L1: NGワード辞書（単語追加はここ）
    l2_patterns.tsv           ← L2: 差別的トロープ（正規表現パターン追加はここ）
    l3_patterns.tsv           ← L3: 悪意ある表現（正規表現パターン追加はここ）
    l4_false_positives.tsv    ← L4: 誤検知除外（除外パターン追加はここ）
    l5_manual.tsv             ← L5: 辞書外マニュアル検知（追加はここ）
    censored_patterns.tsv     ← 伏字検知パターン（追加はここ）
  channel-rules.md            ← チャンネル固有ルール（人間向けドキュメント）
  ng-dictionary.md            ← 8層辞書の説明（人間向けドキュメント）
  youtube-6policy.md          ← YouTube公式ポリシー判定基準
  replacement-db.md           ← 言い換えデータベース
```

**CRITICAL**: scan_v4.py が実際に読むのは `refs/data/*.tsv` のみ。
`refs/ng-dictionary.md` と `refs/channel-rules.md` は人間向けのドキュメントであり、
scan_v4.py は参照しない。TSVファイルが唯一の正（Single Source of Truth）。

### TSVフォーマット仕様

全TSVファイル共通ルール：
- エンコーディング: UTF-8
- 区切り文字: タブ（\t）
- コメント行: `#` で始まる行
- 空行: 無視される

**l1_words.tsv（NGワード追加用）**
```
# カテゴリ	単語/パターン	重大度	タイプ
CH固有禁止	アホ	🔴	plain
CH固有禁止/死	死ぬ	🔴	regex
```
- タイプ `plain`: 単純文字列マッチ（自動でエスケープ）
- タイプ `regex`: 正規表現パターン
- 重大度: 🔴（要修正）/ 🟡（要注意）/ ⚪（監視）

**l4_false_positives.tsv（誤検知除外追加用）**
```
# トリガーワード	除外正規表現
カス	ダマスカス
__DEATH__	必死
```
- トリガーワード: L1で検知された単語名に一致するもの
- `__DEATH__`: 死関連エントリ専用の特殊キー

### ワード追加手順

**新しいNGワードを追加する場合：**
1. `refs/data/l1_words.tsv` を開く
2. 該当カテゴリの末尾に1行追加（タブ区切り）
3. **表記揺れも全て登録する**（漢字・ひらがな・カタカナ）
   例: バカを追加するなら「バカ」「ばか」「ﾊﾞｶ」を全て登録
4. 誤検知が起きる可能性がある場合は `refs/data/l4_false_positives.tsv` にも除外パターンを追加
   例: 「シヌ」を追加 → 「シヌクレイン」を除外パターンに追加

**新しい差別パターンを追加する場合：**
1. `refs/data/l2_patterns.tsv` または `refs/data/l3_patterns.tsv` を開く
2. 末尾に正規表現パターンを1行追加

**新しい誤検知除外を追加する場合：**
1. `refs/data/l4_false_positives.tsv` を開く
2. 末尾に「トリガーワード\t除外パターン」を1行追加

**新しい伏字パターンを追加する場合：**
1. `refs/data/censored_patterns.tsv` を開く
2. 末尾に正規表現パターンを1行追加

### 表記揺れルール（ひらがな・カタカナ・漢字）

NGワードは **漢字・ひらがな・カタカナの全表記** を個別エントリとして登録する。
YouTube AIはどの表記でも同じ単語として認識するため、すり抜けを防ぐ。

例：
```
CH固有禁止	バカ	🔴	plain
CH固有禁止	ばか	🔴	plain
CH固有禁止	馬鹿	🔴	plain
```

ただし、ひらがな・カタカナ表記が別の用語の一部になる場合は、
必ず `l4_false_positives.tsv` に除外パターンを登録する。

例：「しぬ」→「シヌクレイン」が誤検知される
```
# l4_false_positives.tsv
しぬ	シヌクレイン
シヌ	シヌクレイン
```
