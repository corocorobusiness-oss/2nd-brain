# AI開発依頼ヒアリングモード 天才会議 2026-07-02

## テーマ

ユーザーが `AI開発依頼テンプレ_完全版.md` を毎回開いて手入力する代わりに、自然な会話からAIが不足情報を質問し、依頼文へ落とし込み、最後に確認してから `ai-development-flow` へ渡す入口を作ってよいか。

## 評価対象案

```text
1. ユーザーが、作りたい仕組み・修正したいこと・アップデートしたいことを自然に話す
2. AIが不足情報だけ質問して掘り下げる
3. 会話内容から AI開発依頼テンプレ_完全版.md の人間用テンプレへ自動入力する
4. 最後に「つまり、こんなことがしたいんだよね？」と確認する
5. ユーザー承認後、その依頼文を ai-development-flow に渡す
6. 以後は ai-development-flow 本体で案件分類・Goal固定・設計・実装へ進む
```

## 採点基準

ファンアウト前に、以下の基準を固定した。

1. 人間の入力負担が減るか
2. 曖昧な話からテンプレ化する際の聞き漏れを防げるか
3. 質問が多すぎて逆に重くならないか
4. 危険案件を止められるか
5. `ai-development-flow` 本体と責務が混ざらないか
6. 最終確認・承認の文言が十分か
7. MVPとして小さく始められるか

補足: `会議の型.md` では本来、採点基準を先に別ファイル保存してから会議に入る。今回はファンアウト前プロンプトで固定し、この会議録へ保存した。次回以降はファイル保存を先に行う。

## 5人の独立レビュー結果

### 1. PM / プロダクト運用視点

- 判定: 条件付き採用
- 良い点: テンプレ手入力が消え、自然会話から依頼文化できる。最後に確認を挟むため丸投げ暴走を防ぎやすい。
- 重大な穴: 「ユーザーが承認したら」が、依頼文承認・実装承認・本番操作承認のどれか曖昧。
- 修正案: 仕組み名は「AI開発依頼ヒアリング補助」。最終確認文は「依頼文を渡す承認であり、実装開始・本番書き込み・外部送信・削除・自動化ONの承認ではない」と明記する。

### 2. SRE / セキュリティ視点

- 判定: 条件付き採用
- 良い点: 人間用テンプレの自動入力は合理的。既存フローには危険トリガー、承認ゲート、秘密情報保存禁止、未完成で止まる条件がある。
- 重大な穴: 危険トリガー検出が未設計。会話中のトークン、Cookie、個人情報、API応答、生ログを保存するリスクがある。
- 修正案: `handoff_mode: plan_only` を明記する。危険トリガー判定は会話直後と投入直前の2回行う。秘密情報は値を保存せず分類だけ残す。

### 3. ワークフロー設計視点

- 判定: 条件付き採用
- 良い点: 前段が依頼書作成補助に徹すれば、本体フローの責務を壊さない。小さく導入できる。
- 重大な穴: 前段AIが分類・Goal固定・設計までやると、本体と責務が混ざる。無限質問化のリスクがある。
- 修正案: 前段は「危険トリガー候補」と「未確定事項」まで。本分類とGoal固定は `ai-development-flow` 本体で再実施する。質問は最大2ラウンド、1回3問まで。

### 4. 反対派 / 費用対効果視点

- 判定: 条件付き採用。ただし現案のままは不採用。
- 良い点: 手入力負担を減らす狙いは正しい。思いつきを開発フローへつなげやすい。
- 重大な穴: 普段の会話が「開発面談」化する。完全版テンプレの全項目を埋める前提は重い。テンプレ変更に追従する保守負債もある。
- 修正案: MVPは「完全版の自動入力」ではなく「短い開発ブリーフへの圧縮」に絞る。明示トリガー時だけ発動する。5回だけ試し、効果が薄ければ廃止。

### 5. QA / 完成判定視点

- 判定: 条件付き採用
- 良い点: 自然文相談から始められ、質問で不足情報を補う方針は「推測で埋めない」ルールと整合する。
- 重大な穴: 各欄が「ユーザー確認済み」「AI要約」「未確認」のどれか区別できない。品質検査表がない。
- 修正案: 生成物は、依頼文ドラフト、質問/回答ログ、欄ごとの根拠表、品質検査表の4点セットにする。曖昧な「OK」「いいよ」は本番操作承認として扱わない。

## 統合結論

5人全員が **条件付き採用**。

ただし、元案のまま「完全版テンプレを自動入力して、承認後そのまま開発に進む」は危ない。

採用するなら、最初は **AI開発依頼ヒアリング補助** として小さく始める。

正しい位置づけ:

```text
本体フローの前に置く、自然会話を安全な開発依頼ブリーフへ整える受付レイヤー。
実装開始・本番操作・外部送信・削除・自動化ONの承認者ではない。
```

## 採用するMVP

### 発動条件

明示トリガーがある時だけ発動する。

例:
- 「これ開発依頼にして」
- 「仕組み化したい」
- 「ai-development-flowに回したい」
- 「この話を開発テンプレに落として」

雑談、軽い相談、単発の小修正では勝手に発動しない。

### AIがやること

1. 会話を短い開発ブリーフへ圧縮する
2. 判断に影響する不足だけ質問する
3. 質問は最大2ラウンド、1回3問まで
4. 安全項目は推測で埋めない
5. 未確認は `未確認` として残す
6. 危険トリガー候補を明記する
7. 最後に、依頼文として `ai-development-flow` に渡してよいかだけ確認する

### 前段で聞く優先項目

- 作りたいもの
- 困っていること
- できるようにしたいこと
- 触ってよい場所
- 触ってはいけない場所
- 外部送信、認証、個人情報、本番データの有無
- 完成したと言える状態

### 危険トリガーチェック

毎回チェックする。

- 会計、税務、金額確定
- 認証、トークン、Cookie
- 個人情報、秘密情報
- 外部投稿、外部送信、公開範囲変更
- 削除、停止、置換
- launchd / cron / 常駐プロセス / 自動化ジョブ
- 正本ルール、AGENTS.md、CLAUDE.md、業務フロー変更
- 本番データ書き込み
- 二重実行で損害が出る処理

危険トリガーがある場合、前段では `危険トリガー候補あり` として渡す。本分類と人間承認要否は `ai-development-flow` 本体で再判定する。

### 出力フォーマット

```text
【AI開発依頼ブリーフ】

■ 作りたいもの

■ 背景・困っていること

■ できるようにしたいこと

■ やらないこと

■ 既存の場所・関連ファイル

■ 触ってよい場所

■ 触ってはいけない場所

■ 外部送信・認証・個人情報・本番データ

■ 完成したと言える状態

■ 未確認事項

■ 危険トリガー候補

■ 欄ごとの根拠状態
- ユーザー確認済み:
- AI要約・最終確認待ち:
- 未確認:
- 対象外承認済み:

■ handoff_mode
plan_only
```

### 最終確認文

```text
この依頼文を ai-development-flow に渡して、案件分類と設計に進めていい？

※これは「依頼文を渡す承認」であり、実装開始・本番書き込み・外部送信・削除・自動化ONの承認ではありません。
危険操作が必要になった場合は、その操作ごとに、差分・影響範囲・戻し方を出してから別途確認します。
```

## 不採用にする案

- 雑談や軽い相談でも自動でヒアリングを始める
- 完全版テンプレの全項目を毎回埋めようとする
- `OK` や `いいよ` を本番操作承認として扱う
- 前段AIが案件分類、Goal固定、設計、実装許可まで持つ
- 安全項目をAIが推測で埋める
- 未確認があるのに未確認欄を残さず進める

## 初回試用ルール

最初は5回だけ試す。

記録するもの:
- 質問数
- 手戻りの有無
- 未確認の残り方
- 危険トリガーの見落とし有無
- 本当に手入力より速かったか
- ユーザーが面倒に感じたか

5回試して効果が薄ければ廃止または縮小する。効果がある場合だけ、`AI開発依頼テンプレ_完全版.md` と `ai-development-flow` スキルへ正式反映する。

## 最終判断

採用してよい。

ただし、採用するのは「完全版テンプレ自動入力」ではなく、**AI開発依頼ヒアリング補助 / plan-only受付**。

合言葉:

```text
埋めるAIではなく、埋めてはいけない欄を止めるAIにする。
```
