# ai-development-flow V2 設計レビュー 採点基準 2026-07-02

## テーマ

`ai-development-flow V2` の移行設計に、設計段階で見落としている問題点、穴、過剰設計、曖昧な完了条件がないかを独立レビューする。

## 固定採点基準

この基準を独立レビュー前に固定する。

1. 安全ゲート
   - 依頼文承認、実装開始承認、危険操作承認が混ざらないか。
   - `OK` / `いいよ` を本番操作承認にしない設計になっているか。
   - 会計、認証、外部送信、削除、自動化ON、正本変更で止まれるか。

2. 移行安全性
   - V1退避、V2反映、V1復旧の順序が安全か。
   - `agent-skills` 正本、Codex symlink、Vault控えの関係が崩れないか。
   - 復旧手順が「書いてある」だけでなく検証条件まであるか。

3. 振る舞い検証
   - `SKILL_VISIBLE=YES` だけで完了扱いにしていないか。
   - 5 dry-runケースが、V2の危険な失敗を本当に拾えるか。
   - 5回試用ログで、便利さと安全性の両方を判断できるか。

4. ユーザー体験
   - 質問が多すぎて、テンプレ手入力より遅くならないか。
   - 明示トリガーなしの雑談で勝手にヒアリングが始まらないか。
   - 未確認を残しつつ、ユーザーが何を承認しているか分かるか。

5. 正本整合
   - `AI開発依頼テンプレ_完全版.md`、`AI開発フロー_標準テンプレ.md`、`SKILL.md`、`openai.yaml`、運用記録の責務が重複しすぎないか。
   - 正本と控えのどちらを直すべきか迷わないか。
   - 完了時に `導入済み.md` / `NOW.md` / `決定事項.md` / 作業ログの状態が矛盾しないか。

6. 過剰設計
   - V2の安全性に必要なゲートと、毎回運用すると重い儀式が分かれているか。
   - 新しい常駐ジョブ、重い会議、過剰な台帳更新を増やしていないか。

7. 実装可能性
   - 次の実装ステップで何を作ればよいか明確か。
   - 未解決事項が、実装前に止まるべきものと後でよいものに分かれているか。

## 期待する成果物

- 独立エージェント別の指摘
- 重要度つきの統合指摘
- 設計書へ反映すべき修正案
- 今すぐ直すべきもの / 実装前でよいもの / 試用時に見るもの の分類
