## 目的
今後、仕組み・サービス・自動化・ツールをAIに作らせる時に、毎回この依頼文を起点にする。

狙いは「丸投げ」ではなく、AIに案件分類、設計、実装、検証、完成判定、保守記録までを、リスクに応じた重さで進めさせること。

合言葉:
**軽いものは軽く、危ないものは止まる。**

## 1. 最初にAIへ渡す依頼文

以下をコピーして使う。

```text
以下の仕組みを作ってください。

この案件は、次の方針で進めてください。

- 実装後の失敗修正は、実装 -> テスト -> 失敗修正 -> 再テストを最大3回までの制限付き修正ループにする
- 軽い案件は軽く進める
- 通常以上はテストと証拠を残す
- 重要/危険案件は、独立レビュー、完成判定、人間承認が必要か判定する
- 証拠なし、未実行、未説明WARN、未確認REQ、人間承認不足は完成扱いにしない

## 期限・優先度

重要/危険案件、または `handoff_mode: plan_only` の場合は、実装や本番操作の前に別途「実装開始承認」を取ってください。
```

## 2. ユーザーが最低限書けばよい欄

時間がない時は、最低限これだけ書く。

```text
作りたいもの:

困っていること:

できるようにしたいこと:

触ってよい場所:

触ってはいけない場所:

外部送信・本番データ・認証・個人情報の有無:

完成したと言える状態:
```

AIは不足情報を推測で埋めない。判断に影響する不足があれば質問する。
通常以上では、AIが初回出力でGoalを1文に固定する。

## 2.1 AI開発依頼ヒアリング補助MVP

祐馬さんが毎回テンプレを開いて手入力しなくてよいように、明示トリガーがある時だけ、AIが会話を短い開発ブリーフへ落とし込む。

即発動してよい例:
- 「これ開発依頼にして」
- 「この話を開発テンプレに落として」
- 「ai-development-flowに回して」

相談導線に残し、希望があればブリーフ化する例:
- 「仕組み化したい」
- 「自動化したい」
- 「これ作れない？」
- 「こういう仕組みどう？」

確認文:

```text
これは相談のまま続ける？ それとも開発依頼ブリーフにまとめて ai-development-flow に回す？
```

雑談、軽い相談、単発の小修正では勝手に発動しない。

AIがやること:
1. 会話を短い開発ブリーフへ圧縮する
2. まず仮ブリーフを出す
3. 判断に影響する不足だけ質問する
4. 質問は最大2ラウンド、1回3問まで
5. 1問に複数の判断を詰め込まない
6. 普通案件は追加質問1ラウンドまで、危険/ブロッカーだけ2ラウンド可
7. 安全項目は推測で埋めない
8. 未確認は `ブロッカー / 後で確認 / 対象外` に分けて残す
9. 危険トリガー候補を明記する
10. 最後に、依頼文として `ai-development-flow` に渡してよいかだけ確認する

前段で優先して聞くこと:
- 作りたいもの
- 困っていること
- できるようにしたいこと
- 触ってよい場所
- 触ってはいけない場所
- 外部送信、認証、個人情報、本番データの有無
- 完成したと言える状態

毎回チェックする危険トリガー:
- 会計、税務、金額確定
- 認証、トークン、Cookie
- 個人情報、秘密情報
- 外部投稿、外部送信、公開範囲変更
- 削除、停止、置換
- launchd / cron / 常駐プロセス / 自動化ジョブ
- 正本ルール、AGENTS.md、CLAUDE.md、業務フロー変更
- 本番データ書き込み
- 二重実行で損害が出る処理

危険トリガーがある場合、前段では `危険トリガー候補あり` として渡す。本分類、Goal固定、人間承認要否は `ai-development-flow` 本体で再判定する。

出力フォーマット:

```text
【AI開発依頼ブリーフ】

■ 作りたいもの

■ 背景・困っていること

■ できるようにしたいこと

■ やらないこと

■ 既存の場所・関連ファイル

■ 触ってよい場所

■ 触ってはいけない場所

■ 外部送信・認証・個人情報・本番データ

■ 完成したと言える状態

■ 未確認事項
- ブロッカー:
- 後で確認:
- 対象外:

■ 危険トリガー候補

■ 今回承認するもの

■ 未確認のまま渡すもの

■ 実装前に必ず確認するもの

■ 欄ごとの根拠状態
- ユーザー確認済み:
- AI要約・最終確認待ち:
- 未確認:
- 対象外（ユーザーが今回は不要と明言したもの）:

■ handoff_mode
plan_only（設計まで。実装・書き込みなし）
```

最終確認文:

```text
この依頼文を ai-development-flow に渡して、案件分類と設計に進めていい？

※これは「依頼文を渡す承認」であり、実装開始・本番書き込み・外部送信・削除・自動化ONの承認ではありません。
危険操作が必要になった場合は、その操作ごとに、差分・影響範囲・戻し方を出してから別途確認します。
```

依頼文承認後の確認返答:

```text
受け取った承認: 依頼文承認のみ
今回承認された実装開始: なし
今回承認された危険操作: なし
危険トリガー候補: あり（具体名）/ なし / 未確認
次に出すもの: 案件分類、Goal、設計、実装計画
```

危険操作承認として有効な最小フォーマット:

```text
危険操作承認:
承認者:
承認日時:
対象操作:
対象ファイル/対象サービス:
差分:
影響範囲:
戻し方:
実行タイミング:
二重実行防止:
```

`OK`、`いいよ`、`承認` だけでは危険操作承認として扱わない。上の情報が1つでも欠ける場合は、最小権限に倒して「依頼文承認以下」として扱う。

不採用にする運用:
- 雑談や軽い相談でも自動でヒアリングを始める
- 完全版テンプレの全項目を毎回埋めようとする
- `OK` や `いいよ` を本番操作承認として扱う
- `handoff_mode: plan_only` の後にAI自己判断で実装へ進む
- 前段AIが案件分類、Goal固定、設計、実装許可まで持つ
- 安全項目をAIが推測で埋める
- 未確認があるのに未確認欄を残さず進める

最初は5回だけ試し、質問数、手戻り、未確認の残り方、危険トリガーの見落とし、本当に手入力より速かったかを確認する。効果が薄ければ廃止または縮小する。

## 3. AI側の進め方

AIは、依頼を受けたら最初に案件分類する。

| 分類 | 進め方 | エージェント編成 |
|---|---|---|
| 軽い | 最小限の確認で実行 | 司令塔兼実行AI 1体 |
| 通常 | 要件、実装、テスト、証拠を残す | 司令塔/実装AI + QA AI |
| 重要 | 設計、異常系、独立レビュー、完成判定まで行う | 司令塔AI + 実装AI + QA AI + 独立レビューAI + 完成判定AI |
| 危険 | 人間承認なしで本番操作しない | 人間オーナー + 司令塔AI + 分類/安全ゲートAI + 設計AI + 権限限定の実装AI + QA AI + SRE/セキュリティAI + 外部独立レビューAI + 完成判定AI + 記録AI + 人間承認 |

分類に迷ったら一段上げる。

## 4. 1ステップごとの実行ルール

毎ステップで最初から最大時間・最大人数を使わない。

原則:
- まず小さく始める
- 判断に影響する不確実性が残った時だけ深掘りする
- 重要/危険案件だけ複数エージェントや独立レビューを使う
- ゴール、完了条件、証拠、打ち切り条件を決めてから実行する
- 通常以上では、実装前にGoal固定を行う
- 失敗修正は最大3回の制限付き修正ループにする

### Goal固定ステップ

```text
通常以上の案件では、最初に「何が達成されたら成功か」を1文で固定する。
重要/危険案件では必須。
目的を変更する場合は、変更理由、影響範囲、承認要否を出してから進める。
```

Goal固定で必ず残すもの:
- Goal
- 完成条件
- やらないこと
- 本番化しない条件

### リサーチステップ

```text
まず15分相当で一次調査する。
明確な結論が出ればそこで止める。
判断に影響する未確認点が残る場合だけ、追加で30〜60分相当の深掘りに進む。
重要/危険案件なら、別コンテキストのレビューAIで反証する。
```

リサーチで必ず残すもの:
- 調べた範囲
- 使った情報源
- わかったこと
- 未確認のこと
- 判断に影響するリスク
- 追加調査が必要か
- 打ち切った理由

### 設計ステップ

```text
まずMVPの設計を出す。
作りすぎラインを明記する。
重要/危険案件では、実装前に設計レビューを入れる。
```

設計で必ず残すもの:
- 正本データ
- 入力/出力
- 保存先
- 権限境界
- 失敗時の止まり方
- 二重実行防止
- ロールバック
- 監視/通知
- やらないこと

### 実装ステップ

```text
小分けに実装する。
一度に大きく変えない。
各変更ごとに確認コマンドと期待結果を残す。
```

実装で必ず残すもの:
- 触ったファイル
- 変更理由
- 実行コマンド
- 終了コード
- 結果
- 差分
- 失敗した場合の対応

### 制限付き修正ループ

```text
実装 -> テスト -> 失敗修正 -> 再テストは最大3回まで。
同じ失敗が2回続いたら、3回目へ進む前に原因分析する。
3回でPASSしない場合は、完成扱いにせず「未完成 / 要確認」で止める。
```

各ループで必ず残すもの:
- ループ回数
- 失敗内容
- 失敗の証拠
- 変更した内容
- 実行コマンド
- 終了コード
- 再テスト結果
- 次の仮説

危険案件では、修正ループを使って人間承認ゲートを越えてはいけない。

### 検証ステップ

```text
正常系だけで終わらせない。
通常以上では異常系を確認する。
重要/危険案件では停止、復旧、ロールバックも確認する。
```

検証で必ず残すもの:
- 要件ID
- テスト内容
- 期待結果
- 実行結果
- 証拠
- PASS / FAIL / WARN承認済み / 対象外承認済み / 未確認

### 完成判定ステップ

```text
実装者の自己評価だけで完成にしない。
要件、テスト、証拠、WARN、未確認事項、承認ログを照合する。
```

完成扱いにしてはいけない状態:
- 証拠なし
- 未実行
- 終了コード不明
- 未確認REQあり
- 未説明WARNあり
- 独立レビュー未実施
- 未対応の重大指摘あり
- 危険案件の人間承認なし
- 秘密情報/個人情報の混入疑い

## 5. 危険案件トリガー

次のどれか1つでも含む場合は、原則「危険」。

- 会計、税務、金額確定
- 認証、トークン、Cookie
- 個人情報、秘密情報
- 外部投稿、外部送信、公開範囲変更
- 削除、停止、置換
- launchd / cron / 常駐プロセス / 自動化ジョブ
- 正本ルール、AGENTS.md、CLAUDE.md、業務フロー変更
- 本番データ書き込み
- 二重実行で損害が出る処理

危険案件では、AIの判定は「完成候補」まで。

人間承認がない限り、次をしてはいけない。
- 本番データ書き込み
- 外部送信
- 自動実行ON
- 既存系の停止/削除/置換
- 会計登録
- 認証変更
- 正本ルール変更

## 6. 外部AI・複数エージェントの使い方

外部AIは、原則として実装担当にしない。

使う目的:
- 独立レビュー
- 反証
- 見落としチェック
- セキュリティ/運用リスク確認

外部AIに渡してよいもの:
- 目的
- 要件
- 採点基準
- 変更差分
- 実行コマンド
- 終了コード
- stdout/stderr要約
- 証拠保存先
- 既知のWARN/スキップ
- 未確定事項

渡してはいけないもの:
- 実装者の自己評価
- 司令塔AIの擁護説明
- 秘密情報
- トークン
- Cookie
- 生ログ
- 未マスクの個人情報
- 未マスクのAPI応答
- 認証済みURL

除染確認なしで外部AIへ渡したレビューは無効。

## 8. 実案件での初回検証ルール

```text
【初回運用レビュー】

案件名:
分類:
実際に使ったエージェント編成:

詰まった点:

重すぎた点:

軽すぎた点:

漏れた点:

よかった点:

次回から変えること:

テンプレ改訂が必要か:
```

同じ詰まり、同じ手戻り、同じ抜け漏れが2回出たら、テンプレ改訂候補にする。

