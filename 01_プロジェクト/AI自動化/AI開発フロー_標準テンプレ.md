# AI開発フロー 標準テンプレ

作成日: 2026-07-01

## 目的
AIに仕組み・サービス・自動化を作らせる時に、設計から実装、検証、運用、保守管理までを一貫して回すための標準テンプレ。

合言葉:
**証拠なし、未実行、スキップ放置、警告放置、独立レビュー未通過、人間承認未取得は未完成扱い。**

---

## 1. 案件分類ルール

最初に必ず分類する。

| 分類 | 対象 | 必須ゲート |
|---|---|---|
| 軽い | 誤字修正、小さな表示調整、単発メモ整理 | 目的、変更内容、確認結果 |
| 通常 | 小さな機能追加、単発スクリプト、既存処理の改善 | 要件、実装計画、テスト、証拠 |
| 重要 | 長く使う仕組み、繰り返し使う自動化、事業フローに関わるもの | 設計レビュー、異常系テスト、独立レビュー、運用記録 |
| 危険 | 会計、認証、外部投稿、削除、自動化ジョブ、正本ルール変更、個人情報/秘密情報 | 人間承認、dry-run、停止/復旧テスト、監視、保守管理 |

危険案件は、AIだけで完成扱いにしない。

---

## 2. AIへの依頼テンプレ

```text
時間はかかっていいので、急いで実装しないで。
この開発は「AI開発フロー 標準テンプレ」に従って進めて。

まず次をやってください。

1. 案件分類
   - 軽い / 通常 / 重要 / 危険 のどれか
   - その理由
   - 省略してよい工程があるなら、省略理由

2. 課題定義・作る価値判定
   - 誰が困っているか
   - 何に困っているか
   - 作らない場合の損失
   - 既存運用、既存ツール、手動対応で済まない理由
   - 成功指標
   - MVPの範囲
   - 作りすぎライン

3. リサーチ
   - ユーザー/業務フロー
   - 技術制約
   - 既存ファイル・既存仕組み
   - 外部API・規約・ライブラリ
   - 過去の失敗履歴
   - 調査範囲と打ち切り条件

4. 要件定義
   - Must / Should / Could
   - やらないこと
   - 未確定事項
   - 何が残っていたら未完成か
   - 受け入れテスト

5. アーキテクチャ・リスク設計
   - 正本データ
   - 保存先
   - 外部依存
   - 権限境界
   - 秘密情報の扱い
   - 失敗時の止まり方
   - 二重実行防止
   - ロールバック
   - 監視と通知

6. 実装計画
   - 小分け工程
   - 触るファイル
   - 確認コマンド
   - 期待結果
   - ロールバック方法

実装に入る前に、設計案・リスク・完成条件を出して。
重要/危険案件なら、実装前に独立レビューを入れて。

実装後は、要件ごとの証拠対応表を埋めてから完成報告して。
証拠なし、未実行、スキップ放置、警告放置、独立レビュー未通過、人間承認未取得は未完成扱いにして。
```

## 3. 要件・テスト・証拠対応表

完成判定では、要件ごとに証拠を紐づける。

| 要件ID | 要件 | 実装箇所 | テスト内容 | 実行結果 | 証拠 | 判定者 | 状態 |
|---|---|---|---|---|---|---|---|
| REQ-001 |  |  |  |  |  |  | 未確認 |
| REQ-002 |  |  |  |  |  |  | 未確認 |
| REQ-003 |  |  |  |  |  |  | 未確認 |

状態は次のどれかにする。

- PASS
- FAIL
- WARN承認済み
- 未確認
- 対象外承認済み

未確認が残っている場合は完成扱いにしない。

## 6. 差し戻し条件

以下が1つでもあれば完成扱いにしない。

- 証拠なし
- 未実行
- 終了コード不明
- stdout / stderr未確認
- スキップテスト放置
- 未説明WARN
- 未解決の重大リスク
- 仕様書と実装のズレ未説明
- 独立レビュー未実施
- 未対応の重大レビュー指摘
- 危険案件の人間承認なし
- 停止/復旧方法なし
- 監視/通知なし
- 保守管理なし
