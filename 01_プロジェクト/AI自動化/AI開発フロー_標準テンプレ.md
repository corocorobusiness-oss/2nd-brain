作成日: 2026-07-01

最終見直し日: 2026-07-01

## 目的
AIに仕組み・サービス・自動化を作らせる時に、設計から実装、検証、運用、保守管理までを一貫して回すための標準テンプレ。

合言葉:
**証拠なし、未実行、スキップ放置、警告放置、独立レビュー未通過、人間承認未取得は未完成扱い。**

---

## 1. 案件分類ルール

最初に必ず分類する。

| 分類 | 対象 | 必須ゲート |
|---|---|---|
| 軽い | 誤字修正、小さな表示調整、単発メモ整理 | 目的、変更内容、確認結果 |
| 通常 | 小さな機能追加、単発スクリプト、既存処理の改善 | 要件、実装計画、テスト、証拠 |
| 重要 | 長く使う仕組み、繰り返し使う自動化、事業フローに関わるもの | 設計レビュー、異常系テスト、独立レビュー、運用記録 |
| 危険 | 会計、認証、外部投稿、削除、自動化ジョブ、正本ルール変更、個人情報/秘密情報 | 人間承認、dry-run、停止/復旧テスト、監視、保守管理 |

危険案件は、AIだけで完成扱いにしない。

### 1.1 分類の強制ルール

分類は自己申告で下げない。迷ったら一段上げる。

次のどれか1つでも含む場合は、原則「危険」に分類する。

- 会計、税務、金額確定、freee等への書き込み
- 認証、トークン、Cookie、個人情報、秘密情報
- 外部投稿、外部送信、削除、公開範囲変更
- launchd / cron / 常駐プロセス / 自動化ジョブの追加・停止・変更
- 正本ルール、AGENTS.md、CLAUDE.md、業務フローの変更
- 本番データ書き込み
- 二重実行で損害が出る処理

危険トリガーがある場合、AIだけで通常/重要へ格下げしてはいけない。格下げする場合は人間承認を必要とする。

### 1.2 軽量モード

全案件にPhase 0〜14を丸ごと要求すると形骸化するため、分類別に必須範囲を変える。

| 分類 | 必須Phase | 独立レビュー | 監視/保守 |
|---|---|---|---|
| 軽い | Phase 0 / 8 / 12 | 不要 | 原則不要 |
| 通常 | Phase 0 / 1 / 3 / 7 / 8 / 12 | 任意 | 必要なら |
| 重要 | Phase 0〜13 | 必須 | 必須 |
| 危険 | Phase 0〜14 + 承認ゲート | 必須 | 必須 |

省略したPhaseは、省略理由を書く。重要/危険案件で必須Phaseを省略する場合は人間承認が必要。

### 1.3 危険案件の承認ゲート

危険案件は、次の各時点で人間承認が必要。

1. 本番データに書き込む前
2. 外部へ投稿・送信する前
3. 自動実行をONにする前
4. 既存系を停止・削除・置換する前
5. 会計登録、認証変更、正本ルール変更を行う前

承認ログには、承認者、日時、対象操作、差分、本番影響、影響範囲、戻し方を書く。

「承認済み」とだけ書いたものは承認扱いにしない。

## 3. 要件・テスト・証拠対応表

完成判定では、要件ごとに証拠を紐づける。

| 要件ID | 要件 | 実装箇所 | テスト内容 | 実行結果 | 証拠 | 判定者 | 状態 |
|---|---|---|---|---|---|---|---|
| REQ-001 |  |  |  |  |  |  | 未確認 |
| REQ-002 |  |  |  |  |  |  | 未確認 |
| REQ-003 |  |  |  |  |  |  | 未確認 |

状態は次のどれかにする。

- PASS
- FAIL
- WARN承認済み
- 未確認
- 対象外承認済み

未確認が残っている場合は完成扱いにしない。

### 3.1 証拠の最低条件

証拠は再検証できる実体に限る。「確認した」「問題なさそう」は証拠ではない。

最低限、次を残す。

- 実行日時
- 実行環境
- 実行コマンド
- 対象ファイル/対象データ
- 終了コード
- stdout / stderr 要約
- 差分
- ログ、スクリーンショット、API応答、成果物の保存先
- 必要なら commit hash / 成果物hash

### 3.2 証拠の保管ルール

証拠には、トークン、Cookie、個人情報、秘密情報、生ログをそのまま保存しない。

必要な場合は、マスク済み要約だけを保存する。

stdout / stderr、スクリーンショット、API応答を保存する前に、秘密情報や個人情報が混ざっていないか確認する。

### 3.3 WARN / スキップ / 対象外ルール

WARN承認済み、スキップ承認済み、対象外承認済みにする場合は、必ず次を書く。

- 承認者
- 承認日時
- 理由
- 影響範囲
- 代替確認
- 再点検日または期限
- 期限切れ時の扱い

危険案件ではAI単独承認を禁止する。

上記が欠ける場合は「未確認」として完成扱いにしない。

このテンプレも保守対象とする。

- 採用後、最初の実案件で必ずこのフローを1回使い、詰まった点、重すぎた点、漏れた点、改善案を完成報告または改善ログに残す

- 事故後は、原因がテンプレの不足ならテンプレを更新する

## 6. 差し戻し条件

差し戻し条件は案件分類別に適用する。

- 軽い案件: 目的、差分、確認結果が揃えば完了可。独立レビュー、監視、保守管理は原則不要。
- 通常案件: 要件、実装、テスト、証拠を必須にする。
- 重要案件: 設計レビュー、異常系テスト、運用記録、再テスト範囲を必須にする。
- 危険案件: 人間承認、dry-run、停止/復旧、監視通知、台帳更新、独立レビューを必須にする。

適用範囲内で以下が1つでもあれば完成扱いにしない。

- 証拠なし
- 未実行
- 終了コード不明
- stdout / stderr未確認
- スキップテスト放置
- 未説明WARN
- 未解決の重大リスク
- 仕様書と実装のズレ未説明
- 独立レビュー未実施
- 未対応の重大レビュー指摘
