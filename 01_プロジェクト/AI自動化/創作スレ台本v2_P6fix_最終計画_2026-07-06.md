```json

      "claim_text": "解説が担当するclaim",

      "key_phrases": ["検出語"],
      "aliases": ["別名"],
      "source_fact_ids": ["fact_..."],
      "c2_type": ["②背景構造"],
      "allowed_prelude": ["名前だけ", "疑問", "短い茶々"],
      "forbidden_zones": ["thread_before_insert", "thread_after_insert", "other_thread", "closing"]
    }
  ]
}
```

保存先はスキル正本ではなく、各runの案件/evalサンドボックス内に置く。例: `eval/p6fix_trials/<theme>/reserved_claims.json`。本番案件フォルダやスキル配下には保存しない。

優先関係: `forbidden_zones` は「同密度の陳述・結論・背景説明」を禁じる指定であり、`allowed_prelude` の名前だけ・疑問・短い茶々・断片リアクションは、挿入前でも許容する。つまり、断片はOK、claimの要旨を言い切るのはNG。

`demand_type` は最低3種:

- `narrative_required`: 本筋説明に必須。原則、解説対象にしない
- `meta_background`: 解説向き。スレは断片まで
- `hybrid`: 本筋にも出るが、解説は史料・制度・研究史など一段深い層だけ担当

## 5. check_claim_leak.py 仕様

入力: `reserved_claims.json`、生成スレ2本、parts、manifest、assembled CSV/MD  
出力: `claim_leak_candidates.json` と人間読みmd

検出対象:

- 解説前のC4先食い
- 解説後のC6同密度反復
- 別スレでの再説明
- closing/opening/bridgeでの解説claim再説明
- `verdict_essence` と同じ抽象結論

機械ショートリストは高再現・低精度でよい。`key_phrases`、`aliases`、`verdict_essence`の近接、同一トークン共起、抽象結論語を広めに拾い、最終判断はLLM/人が `same_density` で濾す。検出漏れを避けることを優先し、誤検知はレビューで落とす。

判定は2段階:

1. 機械で候補列挙
2. LLM/人が `same_density=true/false` を付ける

`same_density=true` が `closing`、解説後の同一スレ、`other_thread`（別スレ）に残る場合はすべて NEEDS_FIX。P6B/P6Cで起きた「別スレ側へ再流入してから締めで回収する」型を塞ぐため、other_thread は例外にしない。自動編集は禁止。

解説前のC4先食いは既存の `qa_check.py --profile v2 --guard-packet` と検出範囲が重なる。優先関係は、既存C4 guardがスレ生成直後の修正ゲート、`check_claim_leak.py` はアセンブル前後の全文バックストップとする。C4先食い候補が両方に出た場合は、既存C4 guardのNEEDS_FIX判定を一次判断にし、leak gateは証跡と見落とし確認に使う。

## 7.5. closing/opening/bridge v2契約

`closing-v2-guide.md` は次の契約を持つ:

- 情報の再説明ゼロ。解説claim、スレの結論、`verdict_essence` を言い直さない
- 新規claimゼロ。締めで新しい史実・構造説明・評価軸を足さない
- 役割は感情の着地、余韻、視聴後の見方の軽い持ち上げ、次回誘導だけ
- `reserved_claims.json` の `verdict_essence` 一覧を negative constraints として受け取り、同じ要旨を避ける
- opening/bridgeも、解説claimの予告・結論先出し・締めと同じ抽象総括をしない

この契約はv2/P6-fix明示時のみ使う。通常の `shime-guide.md` を置換しない。

## 8. qa_check v2 conclusion cluster

`--profile v2 --conclusion-cluster-gate` を追加案。初期はWARN/計測中心。

ここでいう conclusion cluster は、単発の「つまり」「結局」検出ではない。P6でB3採点者splitが起きた原因は、終盤に評価・総括・教訓化レスが束で並び、住人会話ではなく分担ナレのように見えるかどうかだった。したがって、検出単位はレス1本ではなく cluster とする。

クラスタ判定案:

- 終盤ウィンドウ: 各スレの末尾20%または解説挿入後の後続ブロックを重点対象にする
- 近接条件: 5レス以内に評価/総括/教訓/役割分担系ヒットが3件以上、または隣接・1レス飛びで2件以上続く
- 内容条件: `abstract_verdict`、`role_summary`、`lesson_like`、`closing_claim`、`lecture_finish` のカテゴリが複数混ざる
- 出力単位: 個別レスではなく `cluster_id` ごとのショートリスト。開始seq、終了seq、該当レス、カテゴリ内訳、なぜ分担ナレに見えるかを出す
- 判定: 初期はWARN/計測。ただしP6-fix試作中は、盲検B3で拾われそうなclusterを本文修正対象にする

TSV例:

```tsv
rule_id	category	pattern	weight
cc001	abstract_verdict	要するに|つまり|結局|言ってしまえば	2
cc002	role_summary	この話は.*構図|ポイントは.*だった	3
cc003	closing_claim	だから.*というわけ	2
cc004	lecture_finish	ここに.*差が出た|背景には.*がある	3
cc005	lesson_like	現代にも通じる|今にもつながる|見えてくる	2
```

P6-fix中は、WARNを見て締め・後半スレを崩す。v1既定では読まない。
