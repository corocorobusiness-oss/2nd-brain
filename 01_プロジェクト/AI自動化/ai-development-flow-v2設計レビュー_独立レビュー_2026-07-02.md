## P1 指摘

### P1-1. `plan_only` 承認後に、AI自己判断で実装へ進む余地がある

根拠:
- `AI開発依頼テンプレ_完全版.md` には「実装に入ってよいと判断できる場合は進めてください」という文が残っている
- V2設計では依頼文承認と実装開始承認を分けている

問題:
- ユーザーの「依頼文を渡していい」と「作っていい」が混ざる可能性がある。
- `handoff_mode: plan_only` の意味が、本体側で強制停止条件になっていない。

修正案:
- `handoff_mode: plan_only` を受けた本体は、必ず「分類、Goal、設計、実装計画」までで停止する。
- 軽い案件を除き、実装開始には別途「実装開始承認」を必須にする。
- dry-runに「依頼文OK後、実装しない」ケースを追加する。

### P1-2. 危険操作承認として有効な文面が未固定

根拠:
- V2設計には `OK` / `いいよ` / `承認` だけでは危険操作承認にしない、とある
- しかし、何なら危険操作承認として有効かが未固定

問題:
- 曖昧な承認を最小権限へ倒す方針は良いが、危険操作の有効承認形式がない。
- 後から「これは何のOKだったか」を監査しづらい。

修正案:
- 危険操作は次のような対象入りの定型承認だけ有効にする。

```text
危険操作承認:
対象操作:
対象ファイル/対象サービス:
差分:
影響範囲:
戻し方:
実行タイミング:
```

- 承認後に必ず「受け取った承認: 依頼文承認のみ / 実装開始なし / 危険操作なし」のように短く返す。
- 試用ログへ、直前の承認プロンプト、ユーザー返答原文、AIが解釈した承認種別を残す。

### P1-3. V1退避とV2反映を同じ承認で進める設計が危ない

根拠:
- V2設計では、移行実施承認で V1退避、V2反映、Vault控え同期をまとめてよいとしている
- 一方で、人間承認が必要な地点ではそれぞれ別に止まることになっている

問題:
- 退避hash確認前にV2反映へ進む事故が起きる。
- 「退避」と「反映」と「同期」はリスクが違う。

修正案:
- Phase C完了で必ず停止する。
- `V1退避hash PASS` を人間が確認してから、別ゲートでV2反映承認へ進む。
- まとめて承認する場合でも、操作IDごとに対象、差分、影響範囲、戻し方、停止条件を分ける。

### P1-4. V1復旧対象が不完全

根拠:
- V2反映対象に `AI開発依頼テンプレ_完全版.md` と `AI開発フロー_標準テンプレ.md` が含まれる
- 戻し方は主に `SKILL.md` と `agents/openai.yaml` に寄っている

問題:
- `SKILL.md` と `openai.yaml` だけ戻しても、運用正本がV2化していたら旧挙動に戻らない。

修正案:
- V1退避対象に、V1時点の運用正本スナップショットも含める。
- 復旧条件は少なくとも次の4点セットにする。
  - `SKILL.md`
  - `agents/openai.yaml`
  - 運用正本
  - Vault控え/記録
- `manifest.sha256` を作り、対象ファイル、hash、file mode、symlink targetを残す。

### P1-5. dry-run と 5回試用の証拠性・網羅性が不足

根拠:
- 現在のdry-run記録は3ケースのみ
- V2設計では5ケース全PASSを求めている
- 5回試用は件数だけで、内訳の必須条件がない

問題:
- 低リスク5件だけで正式採用できてしまう。
- `OK` 単体テストでは、直前文脈による承認誤解を拾えない。
- dry-run結果が「PASS」中心で、完全な入出力・判定者・環境・証拠保存先が足りない。

修正案:
- dry-runは5ケースそれぞれに、入力、期待応答、必須出力、禁止出力、PASS/WARN/FAIL基準を固定する。
- `OK` テストは以下の4パターンへ分ける。
  - 依頼文承認後のOK
  - 実装開始確認後のOK
  - 危険操作確認後のOK
  - 文脈なしOK
- 5回試用の必須内訳を固定する。
  - 通常案件 1件
  - 危険案件 1件
  - 曖昧承認 1件
  - 触ってよい場所不明 1件
  - 雑談/非発動 1件
- 試用ログへ WARN理由、影響範囲、承認者、再点検日、期限切れ時の扱いを追加する。

### P1-6. `codex-skills-ready` の扱いが曖昧

根拠:
- `agent-neutral-contract.md` では、`~/agent-skills` がスキル正本
- 同ファイルでは、`codex-skills-ready` は旧スナップショット/控えであり、編集の正本ではない
- V2設計では `codex-skills-ready` を同期対象のVault控えとして扱っている

問題:
- 「旧スナップショット」なのか「現行ミラー」なのかが曖昧。
- どちらを直すべきか迷う。

修正案:
- `codex-skills-ready` を今後ライブ控えにするなら、`agent-neutral-contract.md` を先に更新する。
- ライブ控えにしないなら、V2移行では同期対象から外し、hash記録だけにする。

### P1-7. 運用記録の更新ルールがステータス別に分かれていない

根拠:
- V2設計では、台帳、NOW、判断ログ、作業ログを揃えるとしている
- 一方で、NOWは必須更新ではなく、現在地に影響する場合のみ更新とある

問題:
- 試用中、正式採用、復旧時に、どの記録を必ず更新するか迷う。

修正案:
- ステータス別に `更新必須 / リンクのみ / 更新不要` の表を作る。
- 例:
  - `V2試用中`: 試用ログ必須、NOWは現在地に影響する時だけ1行リンク
  - `V2正式採用`: 決定事項と導入済み台帳は必須
  - `V1復旧`: 戻した理由と再挑戦条件を判断ログへ必須

### P1-8. ヒアリング補助ルールの正本が分散している

根拠:
- `AI開発依頼テンプレ_完全版.md` に §2.1 がある
- V2設計にもほぼ同じ内容がある
- `AI開発依頼ヒアリングモード_天才会議_2026-07-02.md` も関連ファイルになっている

問題:
- 片方だけ直して挙動がズレる。
- 設計書が肥大化しやすい。

修正案:
- Phase -1の正本は `AI開発依頼テンプレ_完全版.md §2.1` に寄せる。
- V2設計は参照と差分だけを書く。
- `SKILL.md` には3〜5行の実行に必要な最小ルールだけ置く。

### P1-9. 発動トリガーが広すぎる

根拠:
- V2設計の発動例に「仕組み化したい」がある
- `automation-architect` の自動化相談導線とも重なる

問題:
- 雑談、相談、自動化アイデアで勝手に開発ヒアリングが始まる可能性がある。
- テンプレ手入力より軽くするはずが、会話が重くなる。

修正案:
- 即発動は「開発依頼にして」「テンプレに落として」「ai-development-flowに回して」へ寄せる。
- 「仕組み化したい」「自動化したい」は、必要なら一度だけ「相談のまま？開発依頼にまとめる？」と確認する。

## P2 指摘

- `SKILL_VISIBLE=YES` はV2が読まれた証拠ではない。V2固有文言、`SKILL.md` hash、`plan_only`応答まで確認する。
- V2候補の置き場が未固定。runtime探索されない staging path に置く。
- `legacy/v1_legacy_YYYYMMDD/SKILL.md` が将来の探索で拾われる可能性がある。非探索名または探索されない検証を追加する。
- `handoff_mode: plan_only` は日本語で「設計まで。実装・書き込みなし」と併記する。
- `手入力より速かったか` の比較基準を固定する。
- 設計書作成は重要、V2反映・正本変更・入口差し替えは危険、と分類を分ける。
- `WARNあり` の扱いを、P0/P1/P2/P3ごとに分ける。
- 5ケースの挙動テストとは別に、`openai.yaml` 構文不正、symlink切れ、Vault控えdiff不一致、SKILLだけ更新/openai未更新、復旧後旧挙動のリリース事故テストを追加する。

## P3 指摘

- `v1_legacy_YYYYMMDD` は同日複数回で衝突するため、`YYYYMMDD-HHMMSS` または短いcommit hashを付ける。
- `shasum` は `sha256` 固定にする。
- 人間承認ログには、承認文そのものも残す。
- `対象外承認済み` は、ユーザーが今回は不要と明言したものだけに限定する。
- 試用ログ保存先は「候補」ではなく固定する。

## 結論

V2設計の方向性は良い。

ただし、現状のままV2候補作成や実装承認へ進むのは早い。

特に危ないのは、危険操作を今すぐ実行してしまうことではなく、以下の3つ。

1. `plan_only` の後に、実装開始まで進んでしまうこと
2. V1へ戻したつもりでも、運用正本が戻っていないこと
3. 5回試用やdry-runが、実は危険ケースを網羅していないこと

次の作業は、V2候補文面作成ではなく、まず既存のV2移行設計へP1修正を反映すること。
