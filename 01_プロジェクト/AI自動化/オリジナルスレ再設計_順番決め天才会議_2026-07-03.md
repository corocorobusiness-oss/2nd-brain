# オリジナルスレ再設計 順番決め天才会議 2026-07-03

## 論点

Fable Five にオリジナル台本を1本作らせて比較するのが先か、neta-forge などリサーチ/brief 側を先に改善するのが先か。

## 固定した採点基準

1. 最終的に再現可能なスキル改善へ落ちるか。
2. neta-forge の構成/brief と creative-thread-pipeline の生成が噛み合うか。
3. Fable Five の強みを活かせるか。
4. 1本だけ良い台本で終わらず、仕組みに還元できるか。
5. 現行パイプラインとの dry-run 比較ができるか。
6. 通常制作の速度を壊さないか。
7. 手戻りが少ないか。
8. AI感/解説重複/前後半重複/dic/規約チェックまで改善に波及するか。

## 参照した現行構造

- neta-forge: ネタ発掘から brief.json まで。M3 fact-lock / M4 structure / M5 gloss-scout / M6 brief-emit が下流へ刺さる。
- creative-thread-pipeline: brief.json を STEP1 で読み、creative-thread-gen / youtube-script-parts / checker / dic / third-party review を呼ぶ。
- 会議の型: 採点基準固定、独立ファンアウト、反対派、統合。

## 5人会議の要約

### 1. リサーチ先行派

- Fable Five に台本を書かせる前に、neta-forge が出す brief.json が「毎回いい台本を生む契約」になっているかを見るべき。
- ただし、neta-forge 全体をいきなり大改修するのではなく、brief.json 契約だけを先に監査する。
- brief に reply_map / after_seq など本文生成後の責務まで入れるのは避ける。

### 2. 台本ベンチ先行派

- Fable Five に1本作らせる価値はある。
- ただし目的は「良い台本をもらう」ではなく、現行パイプラインが再現できていない作劇判断をあぶり出すこと。
- 同じ brief.json、同じ制約、手磨きなしで比較し、差分は必ず brief項目 / prompt規約 / 採点ゲート / dry-runテストへ落とす。

### 3. 折衷設計派

- 最初に固定するのはリサーチ全改善ではなく、同じ brief.json で比較できる最低限の土台。
- その後、Fable台本ベンチを作り、差分をスキル改善へ還元し、neta-forge / pipeline 契約へ戻す。
- Fable は「名作を1本作る先生」ではなく、現行パイプラインを毎回底上げする差分を見つける外部レビュアーとして使う。

### 4. 反対派/費用対効果

- いきなりフル台本生成、リサーチ大改修、全体再構築は避ける。
- 1本だけ良い台本が出ても、再現不能なら当たりガチャで終わる。
- 最初は反対レビューと最小dry-run設計に限定し、勝った部分だけ局所採用する。

### 5. QA/実験設計

- まず現行 vs 新案を同じ物差しで比べる評価設計を作る。
- brief品質、台本A/B、下流波及、Codex実装可能性まで採点する。
- ハードゲートは現行より悪化ゼロ。規約赤、dic漏れ、第三者ゲート赤は0件必須。

## 統合結論

いきなり neta-forge 全体を直すのは重すぎる。逆に、いきなり Fable Five に自由台本を書かせると、良い1本が出ても仕組みに戻らない。

最適順は、次の「brief契約先行のA/B設計」。

1. Codex側で比較条件を固定する。
2. Fable Five に評価設計を作らせる。
3. Fable Five に brief.json 契約を監査させる。
4. 同じ brief.json で、現行パイプライン版と Fable ベンチ版を作る。
5. 匿名A/Bで差分を抜く。
6. 差分を upstream / downstream に仕分ける。
7. 採用候補だけを ai-development-flow の plan_only 開発依頼に変換する。

## 推奨するFable Fiveへの投げ順

### Round 0: Codex側の準備

- 現行 brief.json を1本選ぶ。
- 現行パイプラインの手修正なし出力を用意する。
- 評価表の初期案を用意する。
- 禁止事項を固定する。

禁止事項:

- Fable生成台本のそのまま採用。
- 現行スキルの即上書き。
- 通常制作への即投入。
- 新常駐ジョブ。
- 巨大DB化。
- 解説講義化。

### Round 1: 評価設計

Fable Five への依頼:

```text
あなたはQA・実験設計者です。
目的は、2ch歴史YouTubeのオリジナルスレ生成パイプライン再設計において、
現行 neta-forge + creative-thread-pipeline と新案を比較できる評価設計を作ることです。

いきなり実装案や台本生成はしないでください。
まず以下を定義してください。

1. brief.json品質評価表
2. 台本A/B比較表
3. PASS/FAIL条件
4. 取るべき証拠
5. Codex実装へ落とすために必要な出力形式

重視項目:
- dry-run比較できるか
- 改善が測定可能か
- AI感、解説重複、前後半重複、dic、規約チェックまで波及するか
- 完了条件が明確か
- Codexがpatch計画に落とせるか
```

### Round 2: brief.json契約監査

Fable Five への依頼:

```text
2ch歴史YouTubeのオリジナルスレ生成パイプラインについて、
まず台本本文は作らず、neta-forge が出す brief.json 契約だけを監査してください。

目的:
1本だけ良い台本を作ることではなく、毎回再現できるスキル改善に落とすことです。

見てほしい対象:
- brief.json契約
- fact-lock
- structure
- gloss-scout
- title/scoring
- market
- trend
- creative-thread-pipeline STEP1との接続

出力:
- brief契約の強い点
- 足りない点
- 過剰な点
- validate_briefで追加すべき検品
- pipeline STEP1で確認すべきこと
- dry-run比較ケース
- 台本生成に進んでよい条件

まだ実装案や台本本文は出さないでください。
```

### Round 3: 同一briefで台本ベンチ

- Fable Five に同じ brief.json から台本ベンチを作らせる。
- 台本本文だけでなく、構成意図、レス設計、解説挿入点、視聴維持の狙いも出させる。
- 現行パイプライン版と匿名A/B比較する。

### Round 4: 差分を仕組みに翻訳

差分を次のどれかに分類する。

- upstream: neta-forge / brief.json / validate_brief に入れるべきもの。
- midstream: creative-thread-gen / creative-thread-pipeline / youtube-script-parts に入れるべきもの。
- downstream: youtube-script-checker / ymm4-dic-generator / script-third-party-review に入れるべきもの。
- reject: Fable単発のモデル芸で、再現ルール化できないもの。

### Round 5: 3本dry-runで採用判断

- 1本だけ良いなら不採用寄り。
- 追加3テーマで現行より安定して良くなったものだけ採用候補。
- 通常制作を止めず、v2は横に作って比較する。

## 採用条件

- ハードゲートは現行より悪化ゼロ。
- 規約赤、dic漏れ、第三者ゲート赤は0件必須。
- 改善点が具体STEPに落ちる。
- 同一 brief.json で v1/v2 dry-run 比較できる。
- 通常回では工程を増やさず、看板回だけ重くできる。
- Codex がどのファイルをどう変え、どう検証するかまで落とせる。

## 最終判断

リサーチツールを「全部先に改善」ではなく、リサーチと台本生成の接続点である brief.json 契約を先に監査する。

そのうえで、同じ brief から Fable 台本ベンチを作らせ、現行パイプラインとの差分を測る。

Fable Five は作家として直接採用するのではなく、現行パイプラインの弱点を見つける計測器兼レビュアーとして使う。
