# AI開発フロー 天才会議 2026-07-01

## テーマ
「時間はかかってもいいので、AIに開発させるものを、設計から完成までミスなく組み上げる」ための標準フローとして、次の案がベストプラクティスかを評価する。

## 評価対象フロー
1. リサーチ
2. 要件定義
3. 仕様書
4. 設計レビュー
5. 実装計画
6. 小分け実装
7. 各段階でテスト
8. 統合テスト
9. 独立レビュー
10. 完成判定
11. 運用記録

## 完成条件案
- 設計書あり
- 実装計画あり
- やらないこと明記
- テスト項目あり
- 実行ログあり
- エラー0
- 未解決TODO 0
- 仕様との差分説明あり
- 独立レビューPASS
- 使い方・止め方・壊れた時の対応あり

## 採点基準
案を見る前に、以下の基準で固定する。

1. 長期安定性: 長く使っても壊れにくい設計になっているか。
2. 学習性: 使うほど改善点・失敗・運用知見が残り、次回に活きるか。
3. 要件の抜け漏れ防止: 作る前に、誰の何の問題を解くか、やらないこと、成功条件が明確になるか。
4. 実装ミス防止: 大きな実装を小分けにし、各段階で検証できるか。
5. 完了判定の強さ: 「できた気がする」ではなく、証拠・ログ・テスト・レビューで完成扱いを決められるか。
6. 運用安全性: 使い方、止め方、壊れた時の検知、ロールバック、台帳化まで含むか。
7. AI運用適性: AIが暴走・作りすぎ・確認漏れを起こしにくい流れか。
8. コスト妥当性: 重要案件では重く、軽い案件では簡略化できる余地があるか。

## 会議結果
## 5人の独立レビュー結果

### 1. PM視点
- 採用可否: 条件付き採用
- 良い点: 完成判定が強く、AIの「動いたっぽい」完成を防ぎやすい。
- 最大の穴: 要件定義の前に「何のために作るか」「成功とは何か」「どこまで作るか」を固める工程が弱い。
- 追加すべきもの: 課題定義、成功指標、スコープ確定、実運用シナリオでの確認。

### 2. アーキテクト視点
- 採用可否: 条件付き採用
- 良い点: 設計、実装、テスト、独立レビュー、運用記録まで入っている。
- 最大の穴: 正本データ、外部依存、権限境界、失敗時の停止点が独立ゲートになっていない。
- 追加すべきもの: アーキテクチャ設計、データモデル、外部依存一覧、権限境界、復旧方針。

### 3. QA視点
- 採用可否: 条件付き採用
- 良い点: 小分け実装、段階テスト、統合テスト、独立レビューまである。
- 最大の穴: 要件、テスト、証拠が1対1でつながっていない。
- 追加すべきもの: 要件ごとの証拠対応表、受け入れテスト、異常系テスト、完成判定者による証拠チェック。

### 4. SRE / 運用・セキュリティ視点
- 採用可否: 条件付き採用
- 良い点: 作る工程の品質は高い。
- 最大の穴: 本番で壊れた時の検知、停止、復旧、通知、権限最小化が弱い。
- 追加すべきもの: 監視、異常通知、停止テスト、ロールバック、最小権限、二重実行防止、定期棚卸し。

### 5. 反対派 / 費用対効果視点
- 採用可否: 条件付き採用
- 良い点: 作る範囲が膨らむこと、完成した気分で止まることを防ぎやすい。
- 最大の穴: 案件の重さ別にフローを変える仕組みがない。軽い案件では儀式化し、重要案件では曖昧なハンコになりうる。
- 追加すべきもの: 案件分類、作る価値判定、省略理由、受け入れテスト先決め、kill判定日。

## 統合結論
このフローは「そのまま標準化」ではなく、**条件付き採用**。

採用条件は次の3つ。

1. **最初に案件分類をする**
   - 軽い
   - 通常
   - 重要
   - 危険

2. **完成判定を「書類あり」ではなく「要件ごとの証拠PASS」に変える**
   - 要件
   - 実装箇所
   - テスト
   - 実行結果
   - 証拠
   - 判定者
   を対応させる。

3. **運用・停止・復旧・監視まで完成条件に入れる**
   作った瞬間ではなく、実際に使い始めて壊れた時まで見て完成とする。

4. **保守管理を独立フェーズにする**
   作った後に誰も見なくなると、依存切れ・API変更・認証切れ・仕様変更・不要化に気づけない。運用記録だけでなく、更新・棚卸し・廃止判断まで管理する。

## 採用版フロー

### Phase 0. 案件分類
軽い / 通常 / 重要 / 危険に分ける。

危険案件:
- 会計
- 認証
- 外部投稿
- 削除
- 自動化ジョブ
- 正本ルール変更
- 個人情報や秘密情報を扱うもの

危険案件はAIだけで完成扱いにしない。人間承認を必須にする。

### Phase 1. 課題定義・作る価値判定
- 誰が困っているか
- 何に困っているか
- 作らない場合の損失
- 既存運用、既存ツール、手動対応で済まない理由
- 成功指標
- MVPの範囲
- 作りすぎライン

### Phase 2. リサーチ
- ユーザー/業務フロー
- 技術制約
- 既存ファイル・既存仕組み
- 競合/事例
- 外部API・規約・ライブラリ
- 過去の失敗履歴

リサーチは無制限にしない。調査範囲と打ち切り条件を書く。

### Phase 3. 要件定義・非対象・受け入れ条件
- Must / Should / Could
- やらないこと
- 未確定事項
- 何が残っていたら未完成か
- 受け入れテスト

### Phase 4. アーキテクチャ・リスク設計
- 正本データ
- 保存先
- 外部依存
- 権限境界
- 秘密情報の扱い
- 失敗時の止まり方
- 二重実行防止
- ロールバック
- 監視と通知

### Phase 5. 仕様書・テスト設計
- 画面/API/ファイル/入力/出力
- データ形式
- 正常系
- 異常系
- 受け入れテスト
- 要件とテストの対応表

### Phase 6. 設計レビュー
通常以上の案件で実施。

重要・危険案件では、実装前に独立レビューを入れる。

### Phase 7. 実装計画
小分け単位ごとに次を書く。
- 変更内容
- 触るファイル
- 確認コマンド
- 期待結果
- ロールバック方法

### Phase 8. 小分け実装・段階テスト
各段階で証拠を残す。
- 実行コマンド
- 日時
- 対象ファイル
- 終了コード
- stdout / stderr
- 差分
- スクリーンショットやAPI応答

### Phase 9. 統合テスト・異常系テスト
- 正常系
- 空データ
- API失敗
- 権限なし
- 重複実行
- 途中失敗
- 停止
- 復旧
- ロールバック

### Phase 10. 試運用 / dry-run / canary
いきなり本番完成にしない。

- 読み取り専用
- テスト出力
- 限定対象
- 初回実行確認

### Phase 11. 独立レビュー
実装者と別コンテキストで行う。

レビュー指摘0を目標にしない。**未対応の重大指摘0**を完了条件にする。

### Phase 12. 完成判定
完成条件:
- 案件分類あり
- 成功指標あり
- やらないこと明記
- 要件とテストの対応表あり
- 実装計画あり
- 実行証拠あり
- 未確定事項0、または承認済み例外のみ
- 未解決の重大リスク0
- 未説明のエラー0
- スキップテスト0、または理由と承認あり
- WARNは理由つきで出荷可判断済み
- 仕様との差分説明あり
- 独立レビューのプロンプト、指摘、対応結果あり
- 停止テスト済み
- 復旧/ロールバック手順あり
- 監視、通知、再点検タイミングあり
- 秘密情報、生ログ、トークンを保存していない確認あり
- 危険案件は人間承認済み

fail-close条件:
**証拠なし、未実行、スキップ放置、未解決、警告放置、独立レビュー未通過、人間承認未取得は未完成扱い。**

### Phase 13. 運用記録・改善ログ
- 使い方
- 止め方
- 壊れた時の対応
- 実行マシン
- 起動方法
- 保存先
- 監視方法
- 最終確認日
- 見直し日
- 廃止条件
- Backlog

自動化ジョブは `01_プロジェクト/AI自動化/導入済み.md` に台帳記録する。

### Phase 14. 保守管理
長く使う仕組みは、完成後も保守対象として扱う。

- 保守担当/確認者
- 最終確認日
- 次回点検日
- 依存関係の更新確認
- API/認証/トークン期限
- 外部サービス仕様変更の影響
- エラーログ/警告ログの棚卸し
- 使われているかどうか
- 不要化した時の廃止条件
- 変更時に再実行するテスト
- 仕様書と実装のズレ確認

保守管理の完了条件:
- 次回点検日がある
- 壊れた時の通知先がある
- 更新・停止・廃止の判断基準がある
- 依存関係と認証期限が見える
- 変更時にどのテストを再実行するか分かる

保守されない仕組みは、完成ではなく「放置リスクあり」として扱う。
