## 4. データ設計（正本と使い捨ての分離）

**設計の背骨: 型（ライブラリ＋選定ルール）はAI側のデータが持つ。YMM4はただの出力先。.ymmp は毎回使い捨て。**
YMM4に仕様変更があっても、直すのは出力層（.ymmp writer）だけで、育てた資産は無傷。

### 4.1 素材ライブラリ（画像本体）
- 置き場: `/Volumes/SSD/YouTube/素材ライブラリ/`（新設）
  - `固有/<正準キー>/` 例: `固有/徳川家康/portrait_01.jpg`
  - `汎用/<カテゴリ>/` 例: `汎用/城/castle_night_01.png`
- 入口は2つ、出口は1つ:
  - 🟢 汎用（侍・城・巻物・矢印・背景）＝事前一括整備。AI生成が本命（ライセンス問題なし・絵柄統一・被らない）、実物資料系のみCommons
  - 🟡 固有（武将肖像・合戦図・古地図）＝事前一括は構造的に不可能。台本に出たら都度取得→必ずラベリングして同じライブラリへキャッシュ→次回は取得しない
- 散在する既存生成画像（`\\Mac\Home\.codex\generated_images` 等）は後段フェーズでコピー移行（元は消さない）

### 4.2 素材台帳 assets_ledger.jsonl（正本・追記型）
- 正本置き場: `~/Projects/youtube/asset-pipeline/assets_ledger.jsonl`（git管理＝差分・ロールバック可）。画像本体はSSD、台帳はgit、の分離
- フィールド:
  - `id`（sha256）/ `canonical_key`（Wikipediaページ名。将来Wikidata QID併記可）/ `kind`（固有|汎用）
  - `tags[]` / `caption`（AIが画像を見て付与。検索補助・自由記述はこちらに隔離）
  - `file_path` / `source`（commons|generated|manual）/ `source_url`
  - `license` / `license_ok`（許可リスト方式）/ `credit_text`（CC-BY系は自動生成必須）
  - `gen_prompt` / `gen_model`（生成画像の絵柄再現用）
  - `used_in[]`（動画名・行番号・日付）/ `rejected[]`（却下フィードバック書き戻し）
- **正準キーの原則（本設計の急所）**: キーは自由記述にしない。台本の固有名詞→Wikipedia日本語版ページ名（リダイレクト解決込み）に正規化してから台帳検索。表記ゆれによるキャッシュミス＝「増えるだけで速くならないライブラリ」を構造的に防ぐ
- 二重登録防止: sha256一致、または（canonical_key＋source_url）一致なら追加せず既存を返す

### 4.3 image_plan.json（案件ごと・中間生成物）
セリフ行番号・正準キー（または汎用タグ）・選定asset id・Frame/Length・レイヤーを保持。CSVの行⇔.ymmp音声アイテムの1:1対応（3ペアで実証済み）を座標系として使う。

## 5. 処理アーキテクチャ（二段方式＋6層）

### 5.0 二段方式（2026-07-03確定: 発音はYMM4・素材はCodex）

**発音（Hatsuon の `/` 区切り）と音声長（VoiceLength）はYMM4に作らせる。Codexはそれを一切触らない。**

```text
【Pass 1: 音声確定（YMM4担当）】
台本CSV（読み辞書=ymm4-dic-generatorで誤読候補を事前に潰す）
 → YMM4に読み込ませる → YMM4が /付き発音・音声長を生成 → .ymmp 保存

【Pass 2: 素材追加（Codex担当・本案件）】
YMM4保存済み .ymmp を入力
 → 音声・発音アイテムは無変更のまま（diffで機械検証）
 → ImageItem だけを追加した新規 .ymmp を出力
```

- 二段にする根拠: ①発音の自動区切りはYMM4の自然な挙動が最も正確 ②画像の表示タイミング（Frame/Length）は保存時に確定するVoiceLengthから取るのが正確（往復方式の実証済み挙動と整合） ③誤読が動画編集後半で発覚する事故を、YMM4取り込み前の読み辞書＋取り込みで前倒しに潰せる
- 読み（誤読潰し）は本案件のスコープ外の**前工程**。接続点は「Pass 1に入れるCSVの品質」で、既存の ymm4-dic-generator／YMM4読み自動化本線が担当

### 5.1 素材パイプライン（6層・Pass 2の中身）

```text
台本CSV＋YMM4保存済み .ymmp
 → ①抽出層   解説パート判定・固有名詞/場面タグ抽出 → 正準キー解決（不能語は未解決リストへ）
 → ②資産層   台帳検索（キャッシュヒット判定）
 → ③取得層   固有: Wikipedia本人ページ直 → ページ内画像 → extmetadataでライセンス機械判定 → fail-close
              汎用: ライブラリ → 無ければAI生成（チャンネル絵柄プロンプト規約に従う）
              ※固有物のAI生成フォールバック禁止（史実素材の捏造防止）
 → ④ラベリング層  AIが画像内容を見て判定（肖像/古地図/合戦図/現代写真/無関係）→ tags/caption 付与 → 台帳登録
 → ⑤計画・出力層  image_plan生成（行対応＋保存済みVoiceLengthからFrame/Length算出）
                  → 既存 .ymmp の音声を無変更のまま ImageItem を追加
                  パス変換（/Volumes/SSD/… → \\Mac\SSD\…。教師データの実例から変換表を抽出）
                  出力は常に新規 .ymmp（連番）。既存上書き禁止
 → ⑥検品層   コンタクトシート（画像＋使用行＋出典＋ライセンス＋確信度）→ 祐馬さん確認 → 却下を台帳へ書き戻し
```

### 5.2 確信度3段階（画像誤認への答え）

画像の意味判定は「画像認識＋ファイル名＋出典ページタイトル＋ライセンス＋台本文脈」の5情報を合わせて点数化し、

- **高確信 → 自動配置**
- **低確信 → 候補としてコンタクトシートで確認待ち**（自動配置しない）
- **ライセンス不明 → 絶対に止める**（候補にも出さない）

初期は自動配置の閾値を高めに置き、「7割自動・危ないところは止まる」から始めてライブラリと却下履歴が育つほど自動率を上げる。

### 5.3 取得元サイトの許可リスト（祐馬さん指定制）

取得元は台帳とは別の設定ファイル（sites_allowlist）で管理し、祐馬さんが追加・削除する。サイトごとに自動化レベルを持つ:

| レベル | 挙動 | 例 |
|---|---|---|
| auto | 検索〜DL〜台帳登録まで自動 | Wikipedia / Wikimedia Commons（API・ライセンス機械判定可） |
| suggest | 検索して候補URL提示まで。DLは祐馬さん操作 | イラストAC・写真AC等（規約上、無人大量DL不可） |
| manual | AIは関与しない。手動投入された素材のラベリングのみ | その他 |

### 5.4 選定ルール

同一正準キーに複数画像がある場合、直近使用履歴を避けてローテーション（マンネリ防止）。却下履歴のある画像は順位を下げる／出さない。
