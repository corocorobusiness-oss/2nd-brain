## 1. 現状マップ — 6本がどう繋がっているか

```
┌─────────────────────────────────────────────────────────┐
│ ① neta-research（ネタ発掘・評価・ストック）                  │
│   WebSearch＋YouTube API → predict_score.pyで予測再生数      │
│   → 実スレURL3本以上をQ列に収集 → スプシA〜Q列に書き込み      │
│   🔄学習: 週次再学習(neta-retrain月9:10)＋予測答え合わせ      │

               │ スプシ（ネタストック）のQ列＝候補スレURL
               ▼
┌─────────────────────────────────────────────────────────┐

│   ※「Q列を起点に評価開始」は①側に書いてあるが…（→分断1）     │
└──────────────┬──────────────────────────────────────────┘
               │ 確定構成（前半スレ・後半スレ・前置き方向性）
               ▼
┌─────────────────────────────────────────────────────────┐
│ ③ youtube-pipeline（6ステップ一気通貫の指揮者）             │
│   STEP1 = ②を呼ぶ                                         │
│   STEP2 = thread_scraper.pyでレス取得 ＋ ④でパーツ生成       │
│   STEP3 = ⑤で3段階チェック（通過まで先に進まない）           │

│   STEP5 = ⑥でYMM4辞書生成                                  │
│   STEP6 = タイトル・サムネ案・概要欄・ハッシュタグ           │
│   🔄学習: 台本執筆ルール.md事前読込／制作ログ→月次ルール昇格  │

└──────┬───────────────┬───────────────┬───────────────────┘
       ▼               ▼               ▼
  ④ script-parts   ⑤ script-checker  ⑥ ymm4-dic-generator
  （冒頭・解説・     （8層スキャン＋     （読み仮名抽出
   前置き・締め）     文脈＋ファクト）     → ymm4_user.dic）
```

**強み（すでに繋がっていて優秀なところ）:**
- 学習ループが3系統も生きている: ①週次再学習＋予測の答え合わせ ②制作ログ→台本ルール昇格（月次） ③スレコーパス蓄積→オリジナルスレ生成
- Q列方式（ネタ段階で実URL確保）で「台本化したいのにスレがない」事故を予防
- チェッカーのTSVデータ駆動設計（ワード追加1行で即反映）
- パイプラインの「取得失敗でもスレ差し替え禁止」「自動削除・追加禁止」など事故防止ルールが明文化済み

## 2. 分断ポイント — 手動・未接続になっている箇所

| # | 分断 | 内容 | 影響 |
|---|------|------|------|
| 1 | **②がQ列を見ない** | ①は「②はQ列起点で評価開始」と書いているが、②自身の手順書（ステップ1〜3）にはスプシ参照が一切ない。ゼロから検索し直す設計のまま | Q列方式の意味が半減。①で集めたURLが死蔵され、二度手間＋「見つからない」再発リスク |
| 2 | **スプシの係が不在** | H列（構成パターン）「②が埋める」と①に書いてあるが②の手順にない。I列（シナリオURL）・G列（ステータス）はどのスキルの手順にも更新ステップがない | ストックの状態が手動更新頼みで腐る。「既存31件棚卸し待ち」の根因はこれ |
| 3 | **最終タイトルにスコアを通してない** | STEP6でタイトル3案を出すが、predict_score.pyにかけていない。予測器があるのに最終タイトルは勘で選んでいる | ネタ名と公開タイトルが乖離すると予測の答え合わせ（T列）の精度も落ちる |
| 4 | **②の中身評価が目視** | 「Chromeで各まとめサイトにアクセスして確認」と手動前提。③にはthread_scraper.pyがあり、レス数・文字数を実測できる | 評価のA/B/Cが感覚値。スクレイパー1回で済む確認に時間がかかる |
| 5 | **進行状態がセッション内だけ** | パイプラインの進捗はTodoWrite（セッション内）のみ。今日のプラン上限停止のように途中で切れると進行が消える | 再開時にどこまでやったか探す手間。制作中チャンネル/スプシに痕跡がない |
| 6 | **ドキュメントの錆び** | ①に存在しないスキル名「2ch-history-script」参照が2箇所／③STEP4に古いMCP ID（`mcp__14a26a55...`）直書き | 新セッションが迷う・将来の事故の種 |

## 3. 強化提案（優先順）

### 提案A: スプシを「進行台帳」に昇格（効果◎・工数小）🥇
ネタストックのスプシを、リサーチ〜公開までの**単一の進行台帳**にする。
- ②の手順に「ステップ0: スプシのQ列・O列を読み、候補URLを起点にする」を正式追加
- ②の最終ステップに「確定構成をH列に書き込む」を追加
- ③の完了時に「I列（シナリオURL）＋G列（ストック→制作中）」を自動更新するステップをSTEP4/6に追加
- 公開済みへの更新は既存のverify_predictions（週次答え合わせ）と連携

### 提案B: 最終タイトルも予測スコアを通す（効果○・工数極小）🥈
- ③STEP6の手順に「タイトル3案を `predict_score.py --stdin` にかけ、予測再生数つきで提案する」を1行追加するだけ
- ネタ名→公開タイトルの言い換えでスコアがどう動くかも見えるようになり、答え合わせ精度も上がる

### 提案C: ②のスレ評価を半自動化（効果○・工数中）
- ②のステップ4を「thread_scraper.pyで候補URLを一括取得→レス数・文字数・取得可否を実測→A/B/C判定の根拠にする」に変更
- 副産物: この時点でスクレイプ結果が出るので、③STEP2-1の取得が実質先取りできる（コーパス蓄積も早まる）

### 提案D: 進行状態の永続化（効果○・工数小）
- ③の各STEP完了時に、該当ネタの制作中チャンネル（#MMDD-ネタ名）へ「STEP2完了✅」と1行報告＋スプシG列を更新
- セッションが切れても、チャンネルとスプシを見れば現在地がわかる（今日の事故の再発対策）

### 提案E: ドキュメント掃除（工数極小）
- ①の「2ch-history-script」→「youtube-pipeline / youtube-script-parts」に修正
- ③STEP4の古いMCP ID直書きを「Google Drive MCP」表記に修正

**おすすめの進め方:** A＋B＋E を先行実装（合計30分程度・既存動作を壊さない追記中心）→ C・D は次バッチ。

## 4. 実装時の注意（自己検証メモ）
- ②③のSKILL.md改修は**追記中心**にし、既存の絶対ルール（自動削除禁止・スレ差し替え禁止等）には触らない
- スプシ列更新の自動化は「書き込み」のみ。**行の削除・並べ替えはしない**（削除事前承認ルール準拠）
- H/I/G列の更新はappendではなくupdate（行特定はネタ名＋A列番号の照合で）
- 改修後は次回パイプライン実走（6/17千利休が候補）で動作検証する
