# 24時間AIエージェント構築計画

作成日: 2026-06-11
状態: Phase 1 稼働中。2026-06-27に会計正本整理を反映

> 2026-06-27追記: 旧「AI帳簿/weekly-kicho」は会計正本freee運用から外し、実機停止済み。以下の当時表現は、現行ではfreee経理監査・weekly-accounting・補助台帳確認として扱う。

## 目的

祐馬さんの「一人社長 + AIエージェント部隊」を、24時間に近い形で安定運用する。

ただし、最初から完全自律化はしない。

壊れにくさを優先し、Second Brainを正本にした少数精鋭のエージェント体制を作る。

## 目指す姿

```text
祐馬さん
  ↓ 指示・承認
Claude Code = 司令塔
  ↓ 実行依頼
Codex = 実行担当
  ↓ 保存・検証
Second Brain = 正本
  ↓ 必要な時だけ
launchd / MCP / ブラウザ / freee経理監査 = 道具
  ↑
Hooks = 暴走防止・強制ルール
```

## フェーズ

### Phase 0: Hooksを設計する

状態: 設計追加。

Hooksは、AIが作業する直前・直後に入る安全装置。

最初に入れる候補:

1. Dangerous Action Guard
   - 削除、外部投稿、会計確定、外部アカウント設定変更を止める
   - 人間承認がある時だけ通す

2. Read Before Write Guard
   - `AGENTS.md`、`CLAUDE.md`、`00_システム/10_Agent/`、`00_UserProfile/` を書く前に最新内容の再読を要求する
   - 並行編集事故を防ぐ

3. Retry Loop Guard
   - 同じツール・同じ対象で短時間に失敗が続いたら止める
   - クレジット消費と暴走を防ぐ

4. Session End Recorder
   - 会話終了時に「Second Brainへ戻すべき学び」を候補化する
   - 自動保存ではなく、必要なら確認して保存する

5. Context Refresh Hook
   - 長時間作業時に引き継ぎメモを作る
   - コンテキスト腐敗を防ぐ

注意:

Hooksの実体は各エージェントの設定フォルダに入ることが多い。
認証情報やランタイム設定はSecond Brainへ丸ごと保存しない。
Second Brainには「移植可能なHook仕様」と「再現手順」だけ保存する。

### Phase 1: Watchtowerを作る

状態: ✅ **稼働開始（2026-06-12 ツバキがlaunchd登録完了。毎朝8:30自動実行）**

**重要な設計転換（2026-06-12）**: Google Drive内のVaultはlaunchd起動プロセスから**読み取り不可**（macOS TCC。他ジョブが動くのはclaude経由でFull Disk Accessがあるため）。そこでWatchtowerを「Vaultコンテンツ監視」→「**ローカルインフラ監視官**」に役割転換した。
- 本体: `~/.claude/scripts/watchtower_local.py`（ローカル＝TCC回避）／ラッパー: `run_watchtower.sh`／plist: `com.korokoro.yuma-watchtower`（毎朝8:30）
- 検査内容: launchd全ジョブの最終終了コード・主要ログの鮮度・処理済み台帳の健全性・あおいプロセス生存
- 出力: ローカルログ（`~/.claude/scripts/watchtower.log`）＋**異常時のみDiscord #レポートへ通知**（正常時は沈黙＝スパム防止）
- Vaultコンテンツ（日誌・売上・YouTube進行）の監視はあおい（claude経由でVault読める）の領分に分離。役割分担が明確化
- 旧Vault版本体（`watchtower/watchtower.py`、検査項目は現代化済み）は参考として残置（削除せず）

まずは「勝手に作業するAI」ではなく、「異常と未処理を見つけるAI」を作る。

名前: Yuma OS Watchtower

型: スケジュール型

候補時間:

- 毎日 8:30
- または毎週月曜 10:00

処理:

1. 今日の日誌があるか確認
2. freee経理・weekly-accounting・補助台帳の状態を確認
3. 自動化台帳と既存ジョブのズレを確認
4. YouTube制作の進行メモを確認
5. 日誌へ「今日見るべきこと」を追記

出力:

- `05_日誌/YYYY-MM-DD.md`
- `06_エージェント運用/30_ヘルスチェック/ヘルスチェックログ.md`

禁止:

- 削除
- 外部投稿
- 会計確定
- SNS投稿
- 新規ジョブ作成

### Phase 2: Daily Operatorを作る

日次の実務を補助する。

型: イベント型 + オンデマンド型

対象:

- 売上記録
- 今日のタスク整理
- 日誌へのメモ追記

基本は「ユーザーが投げたら動く」。

完全自動で動かすより、祐馬さんの入力をトリガーにする。

### Phase 3: Weekly Managerを作る

週次レビュー担当。

型: スケジュール型

処理:

1. 週の売上と帳簿状況を確認
2. YouTube制作の進捗を確認
3. 自動化の停止・異常を確認
4. 来週やるべきことを3つに絞る
5. Second Brainへ週次メモを保存

### Phase 4: YouTube Pipeline Agentを強化する

型: オンデマンド型

対象:

## 最初に実装する候補

【自動化提案: Yuma OS Watchtower】

■ 結論

この作業は「スケジュール型」がベスト。

毎朝または週1で、Second Brainと既存自動化の健康状態を確認するだけなので、常時AI判断は不要。

■ 仕組みの全体像

launchd
→ Watchtowerスクリプト
→ Second Brainの各ファイル確認
→ 日誌または `06_エージェント運用/30_ヘルスチェック/ヘルスチェックログ.md` へ追記

■ 実装プラン

1. `01_プロジェクト/AI自動化/watchtower/` を作る
2. チェックスクリプトを作る
3. launchdで定期実行する
4. `導入済み.md` に台帳記録する
5. 1週間運用してから対象を増やす

■ ユーザーの手間

最初に実行時間を決めるだけ。

■ 効果

- ゾンビジョブを早く見つけられる
- 日誌・freee経理・YouTube進行の見落としが減る
- 「AI部隊が動いているか」を毎日見える化できる

■ 注意

新しい自動化ジョブなので、実装前に祐馬さんの承認が必要。
