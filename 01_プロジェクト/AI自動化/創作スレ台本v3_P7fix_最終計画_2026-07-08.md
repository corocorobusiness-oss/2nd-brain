## 5. 実装方針

### A. v3 sidecar選定を「要旨一致」寄りに強化

対象: `select_sidecar_claims_v2.py`

追加する考え方:

- v3では `review_reasons` がある full explanation 候補を自動selectedにしない
- `recommendation: candidate` は、語句一致なし・要旨一致なし・レビュー理由なしの候補だけに限定する
- 名前だけヒットは review のまま許容するが、制度機能・因果・目的がスレで説明されている場合は reject
- `same_density_paraphrase` を検出する

検出の入力:

- `claim_text`
- `verdict_essence`
- `key_phrases`
- `thread_should_not_say`
- 新規任意フィールド `forbidden_paraphrases`
- 新規任意フィールド `same_density_tests`

明暦で拾うべき既知型:

- claim: `火の通り道を切る`
- thread: `火の切れ目になる`
- claim: `空地や広い道`
- thread: `空白が仕事する`
- claim: `防火の余白`
- thread: `余白/空白が味方になる`

実装は完全な意味理解ではなく、高再現・低精度の候補抽出でよい。v3では「危なそうなら自動採用しない」が正。

### B. v3のselected条件を厳格化

現在:

- `recommendation == candidate` の上位を selected
- `review_reasons` があっても selected されうる

変更案:

- v3 full explanation selected 条件:
  - `recommendation == candidate`
  - `review_reasons == []`
  - `reject_reasons == []`
  - `estimated_c1_overlap_pct <= 20`
  - `semantic_overlap_level` が `none` または `low`
- `review_reasons` がある候補は `manual_review_candidates` に回す
- `manual_review_candidates` だけでは required count を満たした扱いにしない
- selectedが2本未満なら `status: insufficient_clean_candidates` で exit 2

これにより、明暦の `火除地と広小路` のような「人間が見れば説明済み」候補を足りない本数合わせで採用しない。

### C. brief-contractに任意補助フィールドを追加

対象: `brief-contract.md`

`schema_version` は `"1"` のまま維持。v3任意フィールドとして以下を追加する。

```json
{
  "sidecar_candidate_pool": [
    {
      "candidate_id": "s1",
      "label": "火除地と広小路",
      "claim_text": "...",
      "verdict_essence": "...",
      "key_phrases": ["火の通り道を切る", "空地や広い道"],
      "forbidden_paraphrases": [
        "火の切れ目",
        "空白が仕事する",
        "余白が防火になる",
        "広い道が逃げ道になる"
      ],
      "same_density_tests": [
        {
          "claim": "火除地や広小路が火の通り道を切る",
          "reject_if_thread_says": ["火の切れ目", "火が渡りにくい場所", "逃げ道にも火の切れ目"]
        }
      ]
    }
  ]
}
```

既存briefでは省略可能。v1/v2は読まない。
