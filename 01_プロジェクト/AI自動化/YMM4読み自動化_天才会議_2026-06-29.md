# YMM4読み自動化 天才会議 2026-06-29

## テーマ
YMM4自動化STEP1の読み生成について、現案「台本全文から各行のHatsuonを直接作る」を最案とするか。

## 候補案
- A: `ymm4-dic-generator`を強化し、台本全文から読み正本を作り、行ごとのHatsuonまで生成する
- B: `.dic`だけを強化し、YMM4側の辞書適用に寄せる
- C: 現行の`reading_fix.py`中心で進め、dicは補助に留める
- D: 一度YMM4に読ませて保存し、YMM4生成のHatsuonを回収・再利用する

## 採点基準
1. 読みの正確性: 固有名詞・歴史語・多読み語・2ch表記をどこまで正しく読めるか
2. YMM4実機適合: `Hatsuon`、`/`区切り、`_`表現、VoiceLength往復方式と相性がよいか
3. 長期運用: 台本ごとに壊れず、使うほど読み資産が育つか
4. 手作業削減: 祐馬さんの耳チェックと編集修正がどれだけ減るか
5. 実装コスト: 今のコード・スキルから無理なく進めるか
6. 失敗時の安全性: 漏れ・誤読・過剰登録・YMM4更新時に止められるか

## 会議ログ
## 5人レビュー結果

### 1. YMM4仕様・実機データ視点
- 判定: A本線 + C実装核 + Dは教師データ回収。Bは補助止まり。
- 根拠: `Hatsuon`空では声が出ないため、`.dic`単独は不可。実物Hatsuonには `/`、`_`、`<NUMK>` があるため、既存YMM4出力から最小ルールを学ぶべき。
- 次の一手: 既存3本の実物HatsuonからYMM4表記化レイヤを作り、家康217行で往復検証する。

### 2. 日本語読み・TTS視点
- 判定: Aを出口にする混合。
- 根拠: `戦/後/行った/上/城/方` などは `.dic` だけでは危険。g2p土台 + dic資産 + flags解決がよい。
- 次の一手: `reading_fix.py` のflagsを位置指定overrideに変換し、未解決flagがあれば `.ymmp` 生成を止める。

### 3. 自動化アーキテクチャ視点
- 判定: A+Cが最案。
- 根拠: `per-video dic`、`reading_overrides`、`per-line context overrides`、`Hatsuon生成物` の4層分離が長期運用に向く。
- 次の一手: 4層ルールを仕様に固定する。

### 4. 制作現場・編集時間視点
- 判定: A主軸 + C実装基盤。
- 根拠: YMM4画面で毎回直すより、台本全文からHatsuonを作り、低確信だけ耳チェックする方が編集時間を減らす。
- 次の一手: 低確信語リストを10語以内に絞る。

### 5. 全力反対派
- 判定: C主軸 + Aの薄い部分だけ採用。
- 反論: `ymm4-dic-generator` にHatsuon生成まで背負わせると二重頭脳になる。`家康_ループテスト.ymmp` はHatsuon入りだが、VoiceLengthは217行すべてゼロで、YMM4保存後の尺回収まで閉じていない。
- 次の一手: Hatsuon生成 → YMM4保存 → VoiceLength全行非ゼロ確認 → tighten までの閉ループ検証を通す。

## 統合結論
最案は「Aを本線、Cを実装核、Dを教師データ、Bを補助」にする混合案。

- A: 台本全文から行ごとのHatsuonを作る方針は採用。
- C: 実際の読み生成エンジンは `reading_fix.py` に寄せる。`ymm4-dic-generator` へ全部背負わせない。
- B: `.dic` は動画ごとの固有名詞・歴史語の補助。`.dic`単独にはしない。
- D: YMM4実物Hatsuonを `/`、`_`、`<NUMK>` の教師データとして使う。YMM4任せ運用にはしない。

## 次の実装方針
1. 4層を固定する: `per-video dic` / `reading_overrides` / `per-line context overrides` / `Hatsuon生成物`
2. `reading_fix.py` のflags未解決時は `.ymmp` 生成停止
3. 実物HatsuonからYMM4表記化レイヤを追加
4. 家康217行で `Hatsuon生成 → YMM4保存 → VoiceLength回収 → tighten` の閉ループを通す
