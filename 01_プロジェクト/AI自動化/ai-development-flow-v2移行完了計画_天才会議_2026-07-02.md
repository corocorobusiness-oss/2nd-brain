# ai-development-flow V2 移行完了計画 天才会議 2026-07-02

## テーマ

`ai-development-flow V2` の設計を、ツール作成までではなく、移行完了・旧版退避・試用・正式化・戻し方・運用記録まで含む形へ組み直す。

## 採点基準

会議前に、以下へ固定した。

- `01_プロジェクト/AI自動化/ai-development-flow-v2移行完了計画_採点基準_2026-07-02.md`

主な基準:
- 移行完了までのフェーズが見えるか
- 「作った」と「移行完了」が分かれているか
- 完了条件が機械的に判定できるか
- 旧版退避と戻し方が安全か
- 人間承認ポイントが明確か
- V2試用ログが残るか
- 失敗時に縮小・廃止・V1復旧へ行けるか
- 正本ファイル間の文言ズレを検知できるか
- 最後に「移行完了」と言い切れるか

## 5人レビュー結果

### 1. PM / 移行責任者

- 判定: 条件付き採用
- 良い点: V2を小さく入れる設計はよい。
- 重大な穴: 「V2を作った」「dry-runした」と「移行完了」が分かれていない。dry-run現物は3ケースで、5ケースに足りない。
- 修正案: `V2作成完了`、`V2試用中`、`V2正式採用・移行完了` を別ステータスにする。

### 2. SRE / リリース安全性

- 判定: 条件付きGO。ただし移行完了とはまだ言えない。
- 良い点: 旧版退避方針と人間承認はある。
- 重大な穴: 試用開始後に誰が何を見て正式化するかが薄い。`SKILL_VISIBLE=YES` は「見える」証拠であって、V2として正しく振る舞う証拠ではない。
- 修正案: Phase A-I に分け、fresh runtime、5 dry-run、5回試用、正式化判定、復旧検証を入れる。

### 3. QA / 完成判定

- 判定: 差し戻し。設計GO候補だが、完成判定としては不足。
- 良い点: V2入口仕様、承認分離、旧版退避、戻し方はある。
- 重大な穴: 5回試用ログの型、差し戻し条件、正式採用判定者が未定義。
- 修正案: 移行状態モデル、試用ログフォーマット、差し戻し条件を設計書本体へ入れる。

### 4. 反対派 / 費用対効果

- 判定: 条件付きGO。
- 良い点: 完了ゲートを足すのは必要。
- やりすぎな点: 会議録やNOW詳細更新を毎回必須にするのは重い。
- 修正案: 承認は細分化しすぎず「移行実施承認」に束ねる。`NOW.md` は試用中が現在地に影響する時だけ更新する。

### 5. Obsidian正本 / 運用記録

- 判定: 条件付きNG。まだ移行完了とは言えない。
- 良い点: plan_only、承認分離、危険トリガー、5回試用は設計に入っている。
- 重大な穴: V2実体がまだスキル正本へ入っていない。試用ログ形式、NOW/台帳/決定事項、戻しテスト証拠が未確定。
- 修正案: 状態表、5回試用ログ、diff/hash/runtime確認、移行完了記録を追加する。

## 統合結論

現状のV2設計は「設計GO」だが「移行完了計画」としては不足。

組み直し方:
- 設計書に `移行ステータス` を追加する
- `V2作成完了` と `V2正式採用・移行完了` を分ける
- dry-runは5ケース全PASSまで試用開始しない
- 5回試用ログのフォーマットを固定する
- `正式採用 / 縮小 / 廃止 / V1復旧` のGo/No-Go基準を固定する
- 正本/控え/台帳/NOW/決定事項/作業ログの文言ズレ検知を入れる
- 旧版復旧手順は、検証証拠がない限り完成扱いにしない

## 採用する組み直し方

重い自動化や常駐ジョブは増やさない。

追加するのはMarkdown上のゲートだけ:
- 移行ステータス表
- 移行フェーズ
- 作成完了条件
- 移行完了条件
- 試用ログフォーマット
- Go/No-Go基準
- 正本ズレ検知
- 移行完了記録テンプレ

## 最終判断

この組み直しを入れれば採用。

ただし、実際の `agent-skills` 編集、V1退避、V2差し替え、台帳/NOW更新は次ステップで人間承認を取ってから行う。
