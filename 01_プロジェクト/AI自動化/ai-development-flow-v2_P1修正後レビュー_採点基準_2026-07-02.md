# ai-development-flow V2 P1修正後レビュー 採点基準 2026-07-02

## テーマ

`ai-development-flow V2` の独立設計レビューで出たP1指摘が、設計書と関連テンプレへ正しく反映されたかを確認する。

## 固定採点基準

この基準を、P1修正後レビュー前に固定する。

1. `plan_only` 停止保証
   - 依頼文承認後に、案件分類・Goal固定・設計・実装計画までで止まるか。
   - AI自己判断で実装開始へ進めないか。

2. 危険操作承認の明確化
   - `OK` / `いいよ` / `承認` だけで危険操作承認にならないか。
   - 対象操作、対象ファイル/サービス、差分、影響範囲、戻し方、実行タイミングが必要になっているか。

3. V1退避とV2反映の停止ゲート
   - V1退避後に必ず停止するか。
   - `V1退避hash PASS` を人間が確認するまでV2反映へ進まないか。

4. V1復旧対象の完全性
   - `SKILL.md` / `agents/openai.yaml` だけでなく、運用正本snapshot/hash、manifest、symlink targetまで復旧設計に入っているか。
   - stagingで復旧リハーサルする設計になっているか。

5. dry-run / 5回試用の網羅性
   - 5ケースの入力、期待応答、必須出力、禁止出力、PASS/WARN/FAIL基準を固定する方針があるか。
   - `OK` 文脈テスト、リリース事故テストが追加されているか。
   - 5回試用の必須内訳が固定されているか。

6. `codex-skills-ready` の扱い
   - `agent-skills` が正本であることが明確か。
   - このV2移行では `codex-skills-ready` をライブ同期しない設計になっているか。

7. 運用記録の更新ルール
   - ステータス別に `導入済み.md` / `NOW.md` / `決定事項.md` / 作業ログ / 試用ログ の更新要否が分かれているか。

8. ヒアリング補助の正本集約
   - Phase -1の正本が `AI開発依頼テンプレ_完全版.md §2.1` に寄っているか。
   - V2移行設計は全文重複ではなく、差分と参照に寄っているか。

9. 発動トリガーの狭さ
   - 即発動トリガーと確認してから発動するトリガーが分かれているか。
   - 雑談、軽い相談、自動化相談で勝手に開発ヒアリングが始まらないか。

10. 次フェーズ判定
   - P0/P1が残らない場合、`V2候補作成へ進める` と言えるか。
   - P0/P1が残る場合、何を直せばよいか具体的に分かるか。

## 判定ルール

- P0: 即停止。V2候補作成不可。
- P1: V2候補作成不可。設計へ再修正が必要。
- P2: V2候補作成は可。ただし候補レビュー時に確認する。
- P3: Backlog可。

## 期待する成果物

- 独立レビュー5視点の結果
- P0/P1/P2/P3の統合判定
- `V2候補作成へ進める / 進めない` の明確な結論
